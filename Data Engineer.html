SQL Patterns
Every question maps to one of 5 patterns — identify the pattern first, then apply the template
Pattern AActivity in a time window — "visited >3 times in 7 days", "purchased once in 30 days"
▼
Keywords: "in last X days", "at least once in", "more than N times". Tool: WHERE date >= TODAY minus X, then GROUP BY + HAVING.
-- Users who visited more than 3 times in last 7 days SELECT user_id, COUNT(*) AS visits FROM visits WHERE visit_date >= CURRENT_DATE - 7 -- the window GROUP BY user_id HAVING COUNT(*) > 3; -- the threshold -- "At least once" = simpler, just DISTINCT SELECT DISTINCT user_id FROM orders WHERE order_date >= CURRENT_DATE - 30;
PySpark
from pyspark.sql.functions import current_date, datediff, count df.filter(datediff(current_date(), col("visit_date")) <= 7) \ .groupBy("user_id").agg(count("*").alias("visits")) \ .filter(col("visits") > 3)
HAVING COUNT(*) > 3 means MORE than 3. HAVING COUNT(*) >= 3 means 3 OR MORE. Know the difference — interviews test this specifically.
Pattern BAbsence / inactivity — "not logged in 3 days", "no order in 6 months"
▼
Mental model: ALL users MINUS users who were ACTIVE in the window. Tool: LEFT JOIN + IS NULL. In PySpark: left_anti join.
-- Users NOT logged in for last 3 days SELECT u.user_id FROM users u LEFT JOIN logins l ON u.user_id = l.user_id AND l.login_date >= CURRENT_DATE - 3 WHERE l.user_id IS NULL; -- NULL = no match = inactive
PySpark — left_anti join
recent = logins_df \ .filter(datediff(current_date(), col("login_date")) <= 3) \ .select("user_id").distinct() # left_anti = keep rows from LEFT with NO match on RIGHT users_df.join(recent, "user_id", "left_anti")
LEFT JOIN + IS NULL equals left_anti join in PySpark. Same result, cleaner syntax. Know both.
Pattern CConsecutive days — "logged in 3 consecutive days", "orders on consecutive days"
▼
The trick: date minus ROW_NUMBER stays the SAME value for consecutive dates. When there is a gap, the value changes. Group by that value and count — if count >= N, you have a streak.
WITH daily AS ( -- Step 1: dedup to one row per user per day (CRITICAL) SELECT user_id, CAST(login_date AS DATE) AS day FROM logins GROUP BY user_id, CAST(login_date AS DATE) ), with_streak AS ( -- Step 2: date minus row_number = same value for consecutive dates SELECT user_id, day, DATE_SUB(day, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY day) ) AS streak_group FROM daily ) -- Step 3: count per streak group, keep >= N consecutive days SELECT DISTINCT user_id FROM with_streak GROUP BY user_id, streak_group HAVING COUNT(*) >= 3;
PySpark
from pyspark.sql.functions import row_number, date_sub, to_date from pyspark.sql.window import Window daily = df.withColumn("day", to_date("login_date")) \ .select("user_id","day").distinct() w = Window.partitionBy("user_id").orderBy("day") daily.withColumn("rn", row_number().over(w)) \ .withColumn("sg", date_sub("day", col("rn").cast("int"))) \ .groupBy("user_id","sg").agg(count("*").alias("days")) \ .filter(col("days") >= 3).select("user_id").distinct()
Step 1 dedup is the most commonly forgotten step. If a user logs in twice on the same day, row numbers break. Always dedup to DATE level first.
Pattern DTop N per group — "top 3 employees per dept", "2nd highest salary"
▼
-- 2nd highest salary per department WITH ranked AS ( SELECT name, dept, salary, DENSE_RANK() OVER ( PARTITION BY dept ORDER BY salary DESC ) AS rnk FROM employees ) SELECT name, dept, salary FROM ranked WHERE rnk = 2; -- Top 3 per department SELECT * FROM ranked WHERE rnk <= 3;
Say this
"I use DENSE_RANK partitioned by department, ordered by salary descending. DENSE_RANK handles ties correctly. I wrap it in a CTE because you cannot filter on a window function in the same SELECT — that is a common mistake."
Pattern ERunning calculations — running total, moving average, cumulative sum
▼
-- Running total SELECT date, sales, SUM(sales) OVER ( ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total FROM daily_sales; -- 7-day moving average SELECT date, revenue, AVG(revenue) OVER ( ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS moving_avg_7d FROM daily_revenue;
PySpark
from pyspark.sql.functions import sum as sum_, avg from pyspark.sql.window import Window # Running total w = Window.orderBy("date").rowsBetween(Window.unboundedPreceding, Window.currentRow) df.withColumn("running_total", sum_("sales").over(w)) # 7-day moving average w7 = Window.orderBy("date").rowsBetween(-6, 0) df.withColumn("moving_avg_7d", avg("revenue").over(w7))
Always specify ROWS BETWEEN explicitly in PySpark. The default RANGE BETWEEN can give unexpected results with duplicate values.

Date Query Patterns
All date-based questions are one of these 3 patterns — label it first
When you see a date question — ask: is it about ACTIVITY (Pattern A), ABSENCE (Pattern B), or CONSECUTIVE days (Pattern C)? That one label tells you exactly which template to use.
CombinedUsers: visited >3 times AND purchased in 30d AND not contacted in 14d
▼
Multiple conditions: break into separate CTEs, one per condition, then JOIN them together.
WITH frequent AS ( SELECT user_id FROM visits WHERE visit_date >= CURRENT_DATE - 7 GROUP BY user_id HAVING COUNT(*) > 3 -- Pattern A ), buyers AS ( SELECT DISTINCT user_id FROM orders WHERE order_date >= CURRENT_DATE - 30 -- Pattern A ), contacted AS ( SELECT DISTINCT user_id FROM contacts WHERE contact_date >= CURRENT_DATE - 14 -- Pattern B (to EXCLUDE) ) SELECT f.user_id FROM frequent f INNER JOIN buyers b ON f.user_id = b.user_id LEFT JOIN contacted c ON f.user_id = c.user_id WHERE c.user_id IS NULL; -- NOT in contacted
PySpark
from pyspark.sql.functions import current_date, datediff frequent = visits.filter(datediff(current_date(),"visit_date")<=7) \ .groupBy("user_id").count().filter(col("count")>3) buyers = orders.filter(datediff(current_date(),"order_date")<=30) \ .select("user_id").distinct() contacted = contacts.filter(datediff(current_date(),"contact_date")<=14) \ .select("user_id").distinct() frequent.join(buyers,"user_id") \ .join(contacted,"user_id","left_anti")

Top AskedEmployees earning more than their department average — the classic window function interview question
▼
WITH dept_avgs AS ( SELECT *, AVG(salary) OVER (PARTITION BY dept) AS dept_avg FROM employees ) SELECT name, dept, salary, ROUND(dept_avg, 0) AS dept_avg, ROUND(salary - dept_avg, 0) AS above_by FROM dept_avgs WHERE salary > dept_avg ORDER BY above_by DESC;
PySpark
from pyspark.sql.window import Window from pyspark.sql.functions import avg w = Window.partitionBy("dept") df.withColumn("dept_avg", avg("salary").over(w)) \ .filter(col("salary") > col("dept_avg"))
Say this
"GROUP BY would collapse all employees into one row per department. Window function calculates the dept average for each row without collapsing. This lets me compare each employee's salary to their department average in one query."

PySpark Core Operations
Operations you use in every pipeline — know these cold
Corecollect()[0][0] — getting a single value out of a DataFrame
▼
PySpark never gives a plain number directly. Everything stays as a DataFrame until you pull it out with collect(). Then index like a nested list.
from pyspark.sql.functions import avg, max, min # Step by step: df.agg(avg("salary")) # still a DataFrame (a table) df.agg(avg("salary")).collect() # list → [Row(avg(salary)=66400.0)] df.agg(avg("salary")).collect()[0] # Row(avg(salary)=66400.0) df.agg(avg("salary")).collect()[0][0] # 66400.0 ← plain float # Multiple aggregates — access by name (safer) result = df.agg( avg("salary").alias("avg_sal"), max("salary").alias("max_sal") ).collect()[0] print(result["avg_sal"]) # access by name not position
Use result["column_name"] not result[0] when getting multiple values. If you reorder aggregations, position indexing silently returns the wrong value.

Top AskedHow do you handle a corrupt file in S3?
▼
from pyspark.sql.functions import col, input_file_name, current_timestamp df = spark.read \ .option("mode", "PERMISSIVE") \ .option("columnNameOfCorruptRecord", "_corrupt_record") \ .schema(schema).json("s3://bucket/raw/") valid_df = df.filter(col("_corrupt_record").isNull()).drop("_corrupt_record") bad_df = df.filter(col("_corrupt_record").isNotNull()) \ .withColumn("source_file", input_file_name()) \ .withColumn("flagged_at", current_timestamp()) valid_df.write.mode("append").parquet("s3://bucket/processed/") bad_df.write.mode("append").json("s3://bucket/quarantine/")
Say this
"PERMISSIVE captures bad rows in _corrupt_record. I split into valid and bad DataFrames — valid to processed, bad to quarantine with source file and timestamp. I alert if bad rate exceeds 5%."
Top AskedSmall files — Glue writing thousands of tiny files, how to fix
▼
# coalesce — reduce partitions, no full shuffle (cheaper) df.coalesce(8).write.parquet("s3://bucket/output/") # repartition — full shuffle, even distribution df.repartition(8).write.parquet("s3://bucket/output/") # Delta OPTIMIZE — compacts files automatically spark.sql("OPTIMIZE delta.`s3://bucket/delta/orders/`")
Say this
"Small files happen because Spark keeps too many partitions after filtering. I use coalesce before writing — no full shuffle so cheaper. Target 128–256MB per file. For Delta I run OPTIMIZE periodically."

Bucket 2 — Performance
Skew · broadcast join · salting · cache vs persist · partitioning
Top AskedSkewed data — detect it and fix it with salting or broadcast join
▼
from pyspark.sql.functions import broadcast, rand, concat, lit, array, explode # Fix 1: broadcast (small table under 50MB) result = df_large.join(broadcast(df_small), "country") # Fix 2: salting (both tables large) SALT = 10 df_large = df_large \ .withColumn("salt", (rand()*SALT).cast("int").cast("string")) \ .withColumn("key", concat(col("country"),lit("_"),col("salt"))) df_small = df_small \ .withColumn("salts", array([lit(str(i)) for i in range(SALT)])) \ .withColumn("salt", explode("salts")) \ .withColumn("key", concat(col("country"),lit("_"),col("salt"))) \ .drop("salts","salt") result = df_large.join(df_small,"key").drop("key","salt")
Say this
"Skew means one partition has far more data — like 80% of rows being country=US. I detect it by checking partition sizes. Broadcast join if one table is small. Salting if both are large — adds a random suffix to spread the hot key across partitions."
ImportantCache vs Persist — and the lazy evaluation trap
▼
from pyspark import StorageLevel df.cache() # = persist(MEMORY_AND_DISK) df.persist(StorageLevel.MEMORY_ONLY) df.persist(StorageLevel.MEMORY_AND_DISK) # PATTERN: cache, trigger, use, unpersist df_active = df.filter(col("status")=="active").cache() count = df_active.count() # TRIGGERS caching (lazy!) agg = df_active.groupBy("dept").count() df_active.unpersist() # FREE memory when done
Say this
"cache() is shorthand for persist with MEMORY_AND_DISK. Cache is LAZY — nothing happens until an action triggers it. I always call .count() right after to force it into memory. I always unpersist when done — uncleaned caches eat RAM for the rest of the job."

Top Asked3 approaches to incremental processing — partition filter, watermark, Glue bookmarks
▼
from datetime import date, timedelta import json, boto3 # Approach 1: partition filter (simplest) today = date.today().strftime("%Y-%m-%d") df = spark.read.parquet("s3://bucket/events/") \ .filter(col("dt") == today) # Approach 2: DynamoDB watermark def get_watermark(job): r = dynamodb.Table("watermarks").get_item(Key={"job_name":job}) return r.get("Item",{}).get("last_ts","1970-01-01") def save_watermark(job, ts): dynamodb.Table("watermarks").put_item(Item={"job_name":job,"last_ts":str(ts)}) last_ts = get_watermark("orders_pipeline") df = spark.read.parquet("s3://bucket/orders/").filter(col("updated_at") > last_ts) # ... process ... write ... new_ts = df.agg(max_("updated_at")).collect()[0][0] save_watermark("orders_pipeline", new_ts) # AFTER successful write # Approach 3: Glue Job Bookmarks # Enable in Glue config: --job-bookmark-option = job-bookmark-enable # Add transformation_ctx="step_name" to every step # End with: job.commit()
Say this
"Three approaches: partition filter for date-partitioned S3, watermark in DynamoDB for tables with updated_at, or Glue Job Bookmarks where AWS manages state. I always save the watermark AFTER a successful write — not before. If I save before and the write fails, those records are skipped forever."

Bucket 4 — Data Quality
4 quality layers · quarantine pattern · fail_reason · alerting
Top AskedSeparate valid and invalid records — the quarantine pattern with fail_reason
▼
from pyspark.sql.functions import when, lit, col, current_timestamp, input_file_name # Tag each bad row with WHY it failed rules = { "null_patient_id": col("patient_id").isNull() | (col("patient_id")==""), "null_encounter": col("encounter_id").isNull(), "future_date": col("admission_date") > current_date(), "negative_bill": col("bill_amount").cast("double") < 0, } fail_condition = lit(None).cast("string") for name, condition in rules.items(): fail_condition = when(condition, lit(name)).otherwise(fail_condition) df = df.withColumn("fail_reason", fail_condition) # Split valid_df = df.filter(col("fail_reason").isNull()).drop("fail_reason") bad_df = df.filter(col("fail_reason").isNotNull()) \ .withColumn("source_file", input_file_name()) \ .withColumn("flagged_at", current_timestamp()) # Show bad breakdown bad_df.groupBy("fail_reason").count().show() # Write each zone valid_df.write.mode("append").parquet("s3://processed/") bad_df.write.mode("append").json("s3://quarantine/") # Alert if bad rate too high if bad_df.count() / df.count() > 0.05: print("ALERT: Bad rate > 5%")
Say this
"I use the quarantine pattern. Each bad row is tagged with a fail_reason column explaining exactly why it failed. Valid records continue. Bad records go to quarantine with source file and timestamp. The fail_reason tells whoever investigates exactly what went wrong."

Top AskedFull Glue job pattern — DynamicFrame for doors, DataFrame for everything inside
▼
from awsglue.dynamicframe import DynamicFrame # Read from Glue Catalog as DynamicFrame dyf = glueContext.create_dynamic_frame.from_catalog( database="my_db", table_name="raw_orders", push_down_predicate="dt >= '2024-01-01'", # filter at read time transformation_ctx="raw_orders" # REQUIRED for bookmarks ) # Convert IMMEDIATELY to DataFrame for all transforms df = dyf.toDF() df = df.filter(col("status")=="active").withColumn("year", year("order_date")) # Convert back ONLY for output dyf_out = DynamicFrame.fromDF(df, glueContext, "output") glueContext.write_dynamic_frame.from_options( frame=dyf_out, connection_type="s3", connection_options={"path":"s3://bucket/processed/"}, format="parquet", transformation_ctx="output" ) job.commit() # CRITICAL — without this bookmarks never advance

Bucket 6 — Lambda
Cold start · timeout · memory=CPU · Lambda detects and delegates
Core PatternProduction Lambda — cold start fix, timeout handling, optimised structure
▼
import boto3, os # OUTSIDE handler — initialised ONCE, reused on warm calls s3_client = boto3.client("s3") glue_client = boto3.client("glue") sns_client = boto3.client("sns") JOB_NAME = os.environ["GLUE_JOB_NAME"] # env vars not hardcoded SNS_TOPIC = os.environ["SNS_TOPIC_ARN"] def lambda_handler(event, context): try: bucket = event["Records"][0]["s3"]["bucket"]["name"] key = event["Records"][0]["s3"]["object"]["key"] if not key.startswith("raw/orders/"): return {"status": "skipped"} response = glue_client.start_job_run( JobName=JOB_NAME, Arguments={"--input_path": f"s3://{bucket}/{key}"} ) return {"status":"triggered","job_run_id":response["JobRunId"]} except Exception as e: sns_client.publish(TopicArn=SNS_TOPIC, Subject="Lambda failure", Message=str(e)) raise # re-raise so CloudWatch also sees the failure
Say this
"Lambda has a 15-minute hard limit so I never do heavy processing there. Lambda detects events and delegates — it spots a new S3 file and triggers Glue. Clients initialised outside the handler run once and reuse on warm calls. I always re-raise after sending the SNS alert so CloudWatch also sees the failure."
1,769 MB is the Lambda memory setting that gives one full vCPU. More RAM = more CPU in Lambda — there is no separate CPU setting. Mentioning this number shows real Lambda knowledge.

Bucket 8 — Pipeline Reliability
Idempotency · quarantine · recovery · Kinesis · CI/CD · alerting
Top AskedJob fails halfway through — and end-to-end pipeline design
▼
Key word: idempotent — running the pipeline twice gives the same result as running it once.
# Idempotent write — overwrite same partition on rerun df.write.mode("overwrite").partitionBy("event_date") \ .parquet("s3://bucket/processed/") # 5-layer pipeline design template # 1. Ingestion: S3 file drop → Lambda trigger → Glue job # 2. Storage: raw/ (CSV) → processed/ (Parquet) → serving/ (Redshift/Athena) # 3. Processing: Glue ETL batch | Spark Streaming real-time # 4. Serving: Athena ad-hoc | Redshift dashboards | DynamoDB app queries # 5. Monitoring: CloudWatch → SNS → team alert within minutes # Recovery playbook # 1. Detect: CloudWatch alarm → SNS alert # 2. Assess: check Glue logs — infra failure or data issue? # 3. Fix: retry (idempotent) OR restoreToVersion (Delta) # 4. Verify: row count source == destination # 5. Prevent: add to unit tests + data quality checks
Say this
"Idempotency is the key — partition overwrite mode means rerunning never creates duplicates. I have two alerting layers: code alerts for business logic failures like bad rate over 5%, and CloudWatch alarms for infrastructure failures. After any failure I fix it AND add it to unit tests so it cannot happen silently again."

System Design Foundations
The 5-step framework and 10 core terms — every SD question uses these
Must KnowThe 5-step framework — say this structure every single time
▼
1. ClarifyAsk BEFORE drawing. Batch or streaming? Volume? Who consumes? SLA for freshness?
2. Estimate scaleDo the math aloud. "500k vehicles × 1 event/30sec = 17,000 events/sec". Determines tools.
3. Design 5 layersIngestion → Storage → Processing → Serving → Monitoring. Walk through each.
4. Trade-offsWhy this over that? Cost vs latency? What changes at 10x scale?
5. Failure handlingJob fails halfway? Corrupt file? Late data? Always address these.
Open every system design answer
"Before I design anything — is this batch or real-time? What is the data volume per day? Who are the downstream consumers? And what is the SLA for data freshness?"
Never start drawing without asking at least 2 clarifying questions. Interviewers deliberately leave requirements vague. Jumping straight to an answer is a red flag.
10 TermsData lake, warehouse, lakehouse, ETL vs ELT, SLA, idempotency, OLAP, Lambda Architecture
▼
Data LakeS3, raw original format, never delete. Your insurance policy — always reprocessable.
Data WarehouseRedshift/Snowflake. Structured, query-optimised, pre-aggregated. Fast for dashboards.
Data LakehouseS3 storage + warehouse features via Delta Lake or Iceberg. Best of both worlds.
ETL vs ELTETL: transform before loading (old). ELT: load raw then transform in warehouse (modern).

S1+S2 — Folder Structure & Multiple Files
zone/sector/feed/dt= · read folder not file · input_file_name() tracking
Concept + CodeWhy dt= format, how to read all files in a folder, and source file tracking
▼
Read the FOLDER, not a specific filename. Spark reads ALL CSV files in that folder and combines them. input_file_name() adds a column showing which file each row came from.
from pyspark.sql.functions import input_file_name, lit, current_timestamp # Read ALL csv files in the folder — not just one file # Hospital A + B + C all land in same dt= folder (Scenario 2) folder = "data/raw/healthcare/admissions/dt=2024-01-15/" df = spark.read \ .option("header", "true") \ .option("inferSchema", "true") \ .csv(folder + "/*.csv") # Add traceability — critical for debugging data issues df = df \ .withColumn("source_file", input_file_name()) \ # WHICH file? .withColumn("ingested_at", current_timestamp()) \ .withColumn("run_date", lit("2024-01-15")) # Check how many rows came from each file df.groupBy("source_file").count().show(truncate=False) # admissions_hospital_a_...csv | 218 # admissions_hospital_b_...csv | 200 # admissions_hospital_c_...csv | 200
Say in interview
"I read the folder path so Spark automatically combines all files. I always add input_file_name() so every row is traceable to its source file. When a data issue is reported weeks later, I can immediately identify which file and which source system sent the bad data."
The dt= prefix is Hive partition format. Glue Crawler auto-registers it as a partition in Glue Catalog. Athena uses partition pruning — reads ONLY matching folders instead of scanning everything.

S3 — Incremental File Processing
Process only NEW data · watermark JSON · 3-day timeline test
Concept + CodeWatermark pattern — read only partitions AFTER last processed date
▼
import json, os from datetime import date, timedelta WATERMARK_FILE = "data/watermarks.json" def get_watermark(sector, feed): if os.path.exists(WATERMARK_FILE): wm = json.load(open(WATERMARK_FILE)) return wm.get(f"{sector}/{feed}", "1970-01-01") return "1970-01-01" # first run — process everything def save_watermark(sector, feed, last_date): wm = json.load(open(WATERMARK_FILE)) if os.path.exists(WATERMARK_FILE) else {} wm[f"{sector}/{feed}"] = last_date json.dump(wm, open(WATERMARK_FILE,"w"), indent=2) # Pipeline run last_date = get_watermark("healthcare", "admissions") print(f"Last processed: {last_date}") # Find all date partitions NEWER than watermark base = "data/raw/healthcare/admissions/" new_partitions = [ p for p in sorted(os.listdir(base)) if p.startswith("dt=") and p.replace("dt=","") > last_date ] print(f"New partitions to process: {new_partitions}") # Read all new partitions dfs = [] for partition in new_partitions: dfs.append(spark.read.option("header","true") .csv(f"{base}/{partition}/*.csv")) if dfs: combined = dfs[0] for df in dfs[1:]: combined = combined.union(df) # ... process combined ... save_watermark("healthcare","admissions", date.today().strftime("%Y-%m-%d"))
Say in interview
"Incremental processing means reading only partitions AFTER the last processed date. I store the watermark in a JSON file locally or DynamoDB on AWS. I save the watermark AFTER a successful write — if I save before and the write fails those records are skipped forever. The pipeline is idempotent — partition overwrite means reruns never duplicate."

S5+S6 — Valid/Invalid Split & Keep Latest Duplicate
fail_reason column · quarantine · ROW_NUMBER dedup
Concept + CodeS5: tag every bad row with fail_reason · S6: ROW_NUMBER to keep latest
▼
Scenario 5 — Validate and quarantine
from pyspark.sql.functions import when, lit, col, current_date rules = { "null_patient_id": col("patient_id").isNull()|(col("patient_id")==""), "future_date": col("admission_date") > current_date(), "negative_bill": col("bill_amount").cast("double") < 0, "invalid_gender": ~col("gender").isin("Male","Female","Other") } fail = lit(None).cast("string") for name, cond in rules.items(): fail = when(cond, lit(name)).otherwise(fail) df = df.withColumn("fail_reason", fail) valid = df.filter(col("fail_reason").isNull()).drop("fail_reason") bad = df.filter(col("fail_reason").isNotNull()) valid.write.mode("append").parquet("data/processed/") bad.write.mode("append").csv("data/quarantine/")
Scenario 6 — Keep latest record on duplicates
from pyspark.sql.window import Window from pyspark.sql.functions import row_number # ROW_NUMBER: partition by business key, order by most recent first w = Window.partitionBy("encounter_id").orderBy(col("ingested_at").desc()) before = df.count() df_deduped = df.withColumn("_rn", row_number().over(w)) \ .filter(col("_rn") == 1) \ .drop("_rn") after = df_deduped.count() print(f"Removed {before-after:,} duplicates")
Say in interview
"I use ROW_NUMBER partitioned by business key ordered by ingested_at descending. Rank 1 is always the most recent record. dropDuplicates keeps an arbitrary row — ROW_NUMBER lets me CHOOSE which row to keep. I always deduplicate BEFORE joining — duplicates multiply when you join first."

Scenario 12 — Merge small files
def merge_small_files(spark, path, target_files=4, threshold_mb=5): # Inventory file sizes small = [f for f in Path(path).rglob("*.parquet") if os.path.getsize(f)/(1024*1024) < threshold_mb] if not small: print("No compaction needed") return print(f"{len(small)} small files found — compacting to {target_files}") df = spark.read.parquet(path) # coalesce: reduces partitions without full shuffle (cheaper) # repartition: full shuffle, even distribution (use if data skewed) df.coalesce(target_files).write.mode("overwrite").parquet(path + "_compacted") # Delta OPTIMIZE (on AWS/Databricks) # spark.sql("OPTIMIZE delta.`s3://bucket/delta/orders/`") # spark.sql("OPTIMIZE delta.`s3://bucket/delta/orders/` ZORDER BY (dept)")
Say in interview
"I partition by low-cardinality columns only — department (7 values) or region (5 values). Partitioning by patient_id would create millions of tiny folders which is far worse than no partitioning. For small files I use coalesce before writing and run Delta OPTIMIZE periodically with ZORDER to both compact files and co-locate related data."
