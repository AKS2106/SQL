Bucket 6 — Lambda

5 questions. Lambda is AWS's serverless compute — runs code without managing servers. In data pipelines Lambda handles triggers, lightweight transforms, orchestration kicks, and alerts. These questions test whether you understand Lambda's limits and how to work around them.
Q1 What is Lambda cold start — and how do you fix it ▼
The core idea — what happens when Lambda runs
Every Lambda function runs inside a container. When the function hasn't been called recently, AWS needs to create a new container from scratch — download your code, start the runtime, initialise everything. That startup time is the cold start — typically 1 to 3 seconds.
Cold start (new container)
Container created from scratch.
Code downloaded and started.
All imports initialised.
Takes 1–3 seconds before your code runs.

Happens after inactivity or when scaling up.
Warm start (reused container)
Container already running from previous call.
AWS reuses it.
Only your handler() runs.
Takes milliseconds.

Happens on subsequent calls within minutes.
Why it matters in data pipelines
If Lambda triggers a Glue job when a file lands on S3 — a 2-second cold start is fine. If Lambda is responding to a real-time API call — 2 seconds is terrible. Know your latency requirement before choosing a fix.
Fix 1 — move heavy work outside the handler
Code outside the handler runs once at container start. Code inside the handler runs on every invocation. Move expensive initialisations outside so they only happen once.
import boto3
import json

# OUTSIDE handler — runs once at cold start, reused on warm calls
s3_client  = boto3.client("s3")          # connection initialised once
glue_client = boto3.client("glue")       # reused across invocations
TABLE_NAME = "pipeline_watermarks"       # constant loaded once

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table(TABLE_NAME)       # table object created once

def lambda_handler(event, context):
    # INSIDE handler — runs on every invocation
    # s3_client and glue_client are already ready — no re-init cost

    bucket = event["Records"][0]["s3"]["bucket"]["name"]
    key    = event["Records"][0]["s3"]["object"]["key"]

    print(f"New file: s3://{bucket}/{key}")

    # Trigger Glue job for the new file
    glue_client.start_job_run(
        JobName="orders_pipeline",
        Arguments={"--input_path": f"s3://{bucket}/{key}"}
    )
    return {"status": "triggered"}
Fix 2 — Provisioned Concurrency (for latency-critical functions)
AWS keeps containers warm and ready at all times. You pay for the reserved capacity even when not used. Only use for functions where cold start latency is unacceptable.
# Configure in AWS Console or Terraform

# Set Provisioned Concurrency = 5  (keeps 5 containers warm)

# Terraform:

  function_name                  = aws_lambda_function.pipeline_trigger.arn
  qualifier                      = aws_lambda_alias.live.name
  provisioned_concurrent_executions = 5
}

Say this in interview
"Cold start is Lambda initialising a new container — takes 1 to 3 seconds. I handle it in three ways depending on the requirement. First, always move heavy imports and client initialisations outside the handler so they run once and are reused. Second, for latency-sensitive functions I use Provisioned Concurrency — keeps containers warm but costs more. Q2 Lambda timeout — how do you handle long-running tasks ▼
Lambda hard limits — know these numbers
Max timeout15 minutes. That is the absolute maximum. Any task that could run longer cannot live in a single Lambda.
Max memory10,240 MB (10GB). Memory also controls CPU — more memory = more CPU proportionally.
Max payload size6MB synchronous, 256KB for SQS events. Large data must go through S3 — never pass it directly in the event.
The wrong way — one big Lambda

Fix 1 — trigger Glue or EMR instead
Lambda should not do heavy data processing. Lambda detects the trigger, starts a Glue job, and returns immediately. Glue handles the actual work with no time limit.
import boto3

glue = boto3.client("glue")

def lambda_handler(event, context):
    # Lambda's job: detect trigger, kick off Glue job, done
    bucket = event["Records"][0]["s3"]["bucket"]["name"]
    key    = event["Records"][0]["s3"]["object"]["key"]

    response = glue.start_job_run(
        JobName="orders_etl_pipeline",
        Arguments={
            "--input_path": f"s3://{bucket}/{key}",
            "--job-bookmark-option": "job-bookmark-enable"
        }
    )
    job_run_id = response["JobRunId"]
    print(f"Glue job started: {job_run_id}")

    # Lambda returns immediately — Glue runs for hours if needed
    return {"status": "started", "job_run_id": job_run_id}
Fix 2 — chain Lambdas via SQS for multi-step work
Break a long task into steps. Each Lambda does one step and puts a message on SQS. The next Lambda picks it up. No single Lambda runs longer than needed.
import boto3, json

sqs = boto3.client("sqs")
QUEUE_URL = "https://sqs.us-east-1.amazonaws.com/123/pipeline-queue"

# Lambda 1: validate file, put message for Lambda 2
def lambda_validate(event, context):
    file_key = event["key"]
    # ... validate ...
    sqs.send_message(
        QueueUrl=QUEUE_URL,
        MessageBody=json.dumps({
            "step": "transform",
            "file_key": file_key,
            "validated": True
        })
    )
    return {"status": "validated"}

# Lambda 2: triggered by SQS, does transform, puts message for Lambda 3
def lambda_transform(event, context):
    for record in event["Records"]:
        body = json.loads(record["body"])
        file_key = body["file_key"]
        # ... transform ...
        sqs.send_message(
            QueueUrl=QUEUE_URL,
            MessageBody=json.dumps({"step": "load", "file_key": file_key})
        )
Fix 3 — AWS Step Functions for complex orchestration
# Step Functions coordinates Lambda chain with:
# - Retry logic (exponential backoff)
# - Error handling per step
# - Parallel execution
# - Visual monitoring of each step

# State machine definition (JSON):
{
  "StartAt": "ValidateFile",
  "States": {
    "ValidateFile": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:...:validate_function",
      "Retry": [{"ErrorEquals":["States.ALL"],"MaxAttempts":3}],
      "Next": "TransformData"
    },
    "TransformData": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:...:transform_function",
      "Next": "LoadOutput"
    },
    "LoadOutput": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:...:load_function",
      "End": true
    }
  }
}
Say this in interview
"Lambda has a 15-minute hard limit so I never put long-running data processing in Lambda. Lambda's role in my pipelines is to detect events and delegate — it spots a new S3 file and immediately triggers a Glue job, then returns. For multi-step workflows I chain Lambdas via SQS messages, or use Step Functions which adds retry logic, error handling, and visual monitoring of each step."
The phrase "Lambda's role is to detect and delegate" is the key mindset. Lambda is a trigger and coordinator in data pipelines — not the engine that does the heavy lifting.
Q3 Memory to CPU ratio in Lambda — and how to optimise ▼
The key fact — Lambda has no separate CPU setting
In Lambda you only set memory. CPU is allocated proportionally to memory automatically — you cannot control CPU independently. More memory = more CPU = faster execution but higher cost per millisecond.
Memory settingvCPU allocationUse case
128 MB0.09 vCPUSimple lookups, lightweight triggers
512 MB0.36 vCPUSmall file processing, API calls
1,769 MB1 full vCPUFirst point of getting a full CPU core
3,008 MB2 vCPUData transforms, parallel processing
10,240 MB6 vCPUMaximum — heavy compute tasks
The counter-intuitive truth — more memory can be cheaper
Lambda charges per GB-second — memory × duration. If doubling memory halves execution time, the total cost stays the same but your function is faster. If it more than halves time, you actually pay less.
# Example: 512MB takes 10 seconds = 5.12 GB-seconds cost
# Example: 1024MB takes 4 seconds = 4.1 GB-seconds cost — CHEAPER and FASTER

# How to find the optimal memory setting
# Use AWS Lambda Power Tuning tool (open source Step Functions state machine)
# It runs your function at multiple memory settings and plots cost vs speed

# In code — check remaining time to detect if nearing timeout
def lambda_handler(event, context):
    # context.get_remaining_time_in_millis() tells you how much time is left
    remaining_ms = context.get_remaining_time_in_millis()
    print(f"Remaining time: {remaining_ms}ms")

    if remaining_ms < 30_000:   # less than 30 seconds left
        # Save progress, send SQS message to continue in next Lambda
        save_checkpoint(current_position)
        trigger_continuation(current_position)
        return {"status": "checkpointed"}
Lambda optimisation checklist
Move init outsideboto3 clients, DB connections, constants — outside handler. Runs once, reused on warm calls.
Right-size memoryTest at 256MB, 512MB, 1024MB. More memory = more CPU. Find the sweet spot for cost vs speed.
Minimise package sizeSmaller deployment package = faster cold start. Use Lambda Layers for large dependencies like pandas.
Use environment variablesConfig like bucket names, table names — env vars not hardcoded. No code change needed to update config.
Avoid large payloadsNever pass large data in the event. Pass S3 paths instead. Lambda reads from S3 when needed.
Say this in interview
"Lambda has no separate CPU setting — memory controls CPU proportionally. At 1,769MB you get one full vCPU. More memory can actually be cheaper because faster execution reduces billable duration. I always move initialisations outside the handler, right-size memory by testing at a few settings, and use Lambda Layers for large dependencies to keep package size small and cold starts fast."
Mentioning 1,769MB as the first full vCPU point is a specific detail that shows real Lambda knowledge. Most people just say "more memory = more CPU" without knowing the threshold.
Q4 Common Lambda patterns in data pipelines — S3 trigger, scheduler, alerting ▼
How Lambda fits into a data pipeline — 3 roles
Event triggerS3 file lands → Lambda fires → triggers Glue job. File-arrival driven pipelines.
SchedulerCloudWatch EventBridge rule fires Lambda on cron schedule → Lambda triggers Glue or Step Functions.
Alert / notificationPipeline fails → Lambda receives CloudWatch alarm → sends SNS, Slack, or email notification.
Pattern 1 — S3 event trigger
# Lambda triggered when file lands in S3
# Configure in Lambda: Add trigger → S3 → bucket → event type: PUT

def lambda_handler(event, context):
    for record in event["Records"]:
        bucket   = record["s3"]["bucket"]["name"]
        key      = record["s3"]["object"]["key"]
        file_size = record["s3"]["object"]["size"]

        print(f"New file: s3://{bucket}/{key} ({file_size} bytes)")

        # Only trigger for expected file types
        if not key.endswith(".json"):
            print(f"Skipping non-JSON file: {key}")
            continue

        # Start Glue job for this file
        boto3.client("glue").start_job_run(
            JobName="raw_ingestion_pipeline",
            Arguments={"--input_path": f"s3://{bucket}/{key}"}
        )
Pattern 2 — scheduled pipeline trigger
# CloudWatch EventBridge cron → Lambda → Glue
# Schedule: cron(0 2 * * ? *)  = every day at 2am UTC

def lambda_handler(event, context):
    from datetime import date
    today = date.today().strftime("%Y-%m-%d")

    boto3.client("glue").start_job_run(
        JobName="daily_orders_pipeline",
        Arguments={
            "--run_date": today,
            "--job-bookmark-option": "job-bookmark-enable"
        }
    )
    print(f"Daily pipeline triggered for {today}")
Pattern 3 — pipeline failure alert
import boto3, json
from datetime import datetime

def lambda_handler(event, context):
    # Triggered by CloudWatch alarm on Glue job failure
    alarm_name  = event["detail"]["alarmName"]
    alarm_state = event["detail"]["state"]["value"]

    if alarm_state != "ALARM":
        return  # only act on ALARM state

    message = f"""
Pipeline Alert - {datetime.utcnow().strftime('%Y-%m-%d %H:%M UTC')}

Alarm: {alarm_name}
Status: FAILED

Action required: Check Glue job logs in CloudWatch.
    """

    boto3.client("sns").publish(
        TopicArn="arn:aws:sns:us-east-1:123456:pipeline-alerts",
        Subject=f"ALERT: {alarm_name}",
        Message=message
    )
Say this in interview
"Lambda plays three roles in my data pipelines. As an event trigger — when a file lands on S3 it fires Lambda which immediately starts the Glue job. As a scheduler — CloudWatch EventBridge fires Lambda on a cron schedule which kicks off the daily pipeline. As an alerter — CloudWatch alarms trigger Lambda when a job fails and Lambda sends SNS notifications to the team. Lambda never does the heavy processing — it coordinates and delegates."
Q5 Lambda optimisation — full checklist for production ▼
The complete optimised Lambda structure
import boto3
import json
import os
from datetime import datetime

# ---- OUTSIDE HANDLER: runs once on cold start, reused on warm ----
# boto3 clients — expensive to create, reuse them
s3_client   = boto3.client("s3")
glue_client = boto3.client("glue")
sns_client  = boto3.client("sns")

# Config from environment variables — not hardcoded
JOB_NAME   = os.environ["GLUE_JOB_NAME"]     # "orders_pipeline"
SNS_TOPIC  = os.environ["SNS_TOPIC_ARN"]      # "arn:aws:sns:..."
BUCKET_OUT = os.environ["OUTPUT_BUCKET"]      # "my-processed-bucket"

# ---- HANDLER: runs on every invocation ----
def lambda_handler(event, context):
    try:
        # Check remaining time — defensive programming
        remaining = context.get_remaining_time_in_millis()
        print(f"Remaining time: {remaining}ms")

        # Parse event
        record = event["Records"][0]
        bucket = record["s3"]["bucket"]["name"]
        key    = record["s3"]["object"]["key"]

        # Validate — skip unexpected files
        if not key.startswith("raw/orders/"):
            print(f"Skipping unexpected key: {key}")
            return {"status": "skipped"}

        # Trigger Glue — Lambda delegates, does not process
        response = glue_client.start_job_run(
            JobName=JOB_NAME,
            Arguments={"--input_path": f"s3://{bucket}/{key}"}
        )

        print(f"Triggered {JOB_NAME}: {response['JobRunId']}")
        return {
            "status":     "triggered",
            "job_run_id": response["JobRunId"]
        }

    except Exception as e:
        # Alert on any failure
        sns_client.publish(
            TopicArn=SNS_TOPIC,
            Subject="Lambda trigger failure",
            Message=f"Error: {str(e)}\nEvent: {json.dumps(event)}"
        )
        raise   # re-raise so Lambda marks invocation as failed
Optimisation summary — all in one place
Clients outsideboto3 clients initialised once — reused on every warm invocation. Single biggest performance gain.
Env variablesNo hardcoded ARNs or bucket names. Change config without redeployment.
Remaining time checkcontext.get_remaining_time_in_millis() — detect if approaching timeout and checkpoint gracefully.
Input validationCheck key prefix before processing — skip unexpected files, avoid unnecessary Glue job triggers.
Try/except with alertAny exception sends SNS alert AND re-raises so Lambda marks the invocation as failed for CloudWatch.
Lambda LayersLarge packages like pandas, numpy — put in a Layer shared across functions. Smaller deployment = faster cold start.
Say this in interview
"My production Lambda pattern: clients and config outside the handler so they initialise once and reuse on warm calls. Config from environment variables so I never hardcode ARNs or bucket names. Input validation to skip unexpected events. Try/except that both alerts via SNS and re-raises so CloudWatch sees the failure. And I check remaining time so I can checkpoint gracefully if approaching the timeout."
Re-raising after SNS alert is a subtle but important detail. If you only send the SNS message and return normally, CloudWatch sees the invocation as successful and no alarm fires. Always re-raise so both alert channels fire.
