Bucket 4 — Data Quality

5 questions. The core idea: bad data is inevitable. Your job is not to prevent it — it is to detect it, separate it, preserve it, and alert on it. Every question here tests whether you think about data defensively.
Q1 What is data quality — the 4 layers every pipeline needs ▼
The core idea
Data quality is not one check — it is four layers, each catching different types of problems. In interviews, listing all four layers immediately signals that you think about data like a senior engineer.
Layer 1 — SchemaDoes the data have the right columns and types? A column that should be integer arriving as string breaks downstream queries silently.
Layer 2 — CompletenessAre required fields NULL? A customer record with no customer_id is useless — cannot join it to anything.
Layer 3 — Business rulesDoes the data make sense? Salary cannot be negative. Age must be 0–120. Order date cannot be in the future.
Layer 4 — Volume / statsDid the expected amount of data arrive? If yesterday had 100k rows and today has 500 — something broke upstream. Alert immediately.
from pyspark.sql.functions import col, count, when, current_date, datediff

def run_quality_checks(df, job_name):
    total = df.count()
    results = {}

    # Layer 1 — schema validated at read time via .schema()

    # Layer 2 — completeness: count NULLs in critical columns
    results["null_user_ids"]  = df.filter(col("user_id").isNull()).count()
    results["null_amounts"]   = df.filter(col("amount").isNull()).count()

    # Layer 3 — business rules
    results["negative_amount"] = df.filter(col("amount") < 0).count()
    results["future_dates"]    = df.filter(
        col("order_date") > current_date()
    ).count()
    results["invalid_status"]  = df.filter(
        ~col("status").isin(["active","inactive","pending"])
    ).count()

    # Layer 4 — volume check (compare to yesterday stored in DynamoDB)
    results["total_rows"] = total

    # Print results
    for check, cnt in results.items():
        status = "PASS" if cnt == 0 else "FAIL"
        print(f"[{status}] {check}: {cnt} rows affected")

    return results
Say this in interview
"I think about data quality in four layers: schema validation at read time, completeness checks for required fields, business rule checks like salary cannot be negative, and volume checks to catch upstream failures. Each layer catches different problems and together they give me confidence the data is trustworthy."
Listing four layers unprompted makes you sound like someone who has dealt with real production data quality issues. Most candidates only mention NULLs.
Q2 How do you separate valid and corrupted records in a JSON file ▼
What corrupted JSON looks like
JSON corruption can mean: broken syntax (missing closing brace), wrong data types, missing required fields, or values that fail business rules. Each needs a different detection method.
Read with PERMISSIVE
→
Structural check (_corrupt_record)
→
Business rule checks
→
Split valid / bad
→
Write to separate zones
from pyspark.sql.functions import (
    col, when, lit, input_file_name,
    current_timestamp, concat_ws
)

# Step 1: Read with PERMISSIVE — broken JSON goes to _corrupt_record
df = spark.read \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .schema(schema) \
    .json("s3://bucket/raw/events/")

# Step 2: Add a reason column — WHY is each row bad
df = df.withColumn("fail_reason",
    when(col("_corrupt_record").isNotNull(),
         lit("broken_json"))
    .when(col("user_id").isNull(),
         lit("null_user_id"))
    .when(col("amount").isNull(),
         lit("null_amount"))
    .when(col("amount") < 0,
         lit("negative_amount"))
    .when(col("event_date") > current_date(),
         lit("future_date"))
    .otherwise(lit(None))   # None means valid
)

# Step 3: Split into valid and bad
valid_df = df.filter(col("fail_reason").isNull()) \
             .drop("fail_reason", "_corrupt_record")

bad_df = df.filter(col("fail_reason").isNotNull()) \
           .withColumn("source_file", input_file_name()) \
           .withColumn("flagged_at",  current_timestamp())

# Step 4: Log the split
total = df.count()
bad   = bad_df.count()
print(f"Valid: {total-bad} | Bad: {bad} ({100*bad/total:.1f}%)")

# Step 5: Write each to separate S3 locations
valid_df.write.mode("append").parquet("s3://bucket/processed/")
bad_df.write.mode("append").json("s3://bucket/quarantine/")
Say this in interview
"I read with PERMISSIVE mode which puts broken JSON into a _corrupt_record column. Then I add a fail_reason column using CASE WHEN — checking for broken JSON, NULL required fields, negative amounts, future dates. I split on that column — valid records go to processed, bad records go to quarantine with the source filename and timestamp. The fail_reason tells whoever investigates exactly why each record failed."
The fail_reason column is what separates a basic answer from a great one. Without it you have a pile of bad records with no idea why they failed.
Q3 Handle duplicate rows while processing in chunks ▼
Two types of duplicates — handle them differently
Within-chunk duplicates
Same record appears twice in the same batch. Fix: dropDuplicates() or ROW_NUMBER within the chunk before writing.
Cross-chunk duplicates
Record in chunk 1 also appears in chunk 2 (e.g. a reprocessed file). Fix: Delta Lake MERGE or check against existing data before inserting.
Fix 1 — dropDuplicates for within-chunk dedup
# Simple dedup — keep one row per unique key
df_deduped = df.dropDuplicates(["order_id"])

# Dedup keeping specific row — e.g. keep the most recent
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number

w = Window.partitionBy("order_id").orderBy(col("updated_at").desc())
df_deduped = df.withColumn("rn", row_number().over(w)) \
               .filter(col("rn") == 1) \
               .drop("rn")
Fix 2 — Delta Lake MERGE for cross-chunk dedup (best for production)
from delta.tables import DeltaTable

target = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")

# MERGE = if record exists update it, if not insert it
# This handles duplicates from any source automatically
target.alias("t").merge(
    df_deduped.alias("s"),
    "t.order_id = s.order_id"          # match on primary key
).whenMatchedUpdateAll() \             # update if already exists
 .whenNotMatchedInsertAll() \          # insert if new
 .execute()
Fix 3 — pandas chunked dedup (for large CSV processing)
import pandas as pd

seen_ids = set()   # track IDs already processed
clean_chunks = []

for chunk in pd.read_csv("large_file.csv", chunksize=100_000):
    # Remove within-chunk duplicates
    chunk = chunk.drop_duplicates(subset=["order_id"])
    # Remove cross-chunk duplicates
    chunk = chunk[~chunk["order_id"].isin(seen_ids)]
    seen_ids.update(chunk["order_id"].tolist())
    clean_chunks.append(chunk)

final = pd.concat(clean_chunks)
Say this in interview
"There are two kinds of duplicates — within a chunk and across chunks. Within a chunk I use dropDuplicates or ROW_NUMBER to keep the most recent record. Across chunks the cleanest solution is Delta Lake MERGE — it upserts so whether a record is new or already exists, the result is always correct. For pandas chunking I maintain a seen_ids set across chunks."
Delta Lake MERGE is the production answer to cross-chunk duplicates. It is worth mentioning even if you also give the dropDuplicates answer — shows you know the modern approach.
Q4 Identify bad records and compute unique users after filtering — S3 logs question ▼
The question from your interview list
"Given S3 logs with user activities, identify bad records and compute how many unique users remain after filtering."
This is a practical version of Q2 — apply the quarantine pattern then count distinct users on the clean data.
from pyspark.sql.functions import (
    col, when, lit, countDistinct,
    input_file_name, current_timestamp, current_date
)

# Step 1: Read S3 logs
df = spark.read \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .schema(schema) \
    .json("s3://bucket/logs/user_activity/")

# Step 2: Tag each row with a reason if it is bad
df = df.withColumn("fail_reason",
    when(col("_corrupt_record").isNotNull(), lit("broken_json"))
    .when(col("user_id").isNull(),           lit("null_user_id"))
    .when(col("event_type").isNull(),        lit("null_event"))
    .when(col("timestamp").isNull(),         lit("null_timestamp"))
    .when(col("timestamp") > current_date(), lit("future_timestamp"))
    .otherwise(lit(None))
)

# Step 3: Split valid and bad
valid_df = df.filter(col("fail_reason").isNull()) \
             .drop("fail_reason", "_corrupt_record")

bad_df = df.filter(col("fail_reason").isNotNull()) \
           .withColumn("source_file", input_file_name()) \
           .withColumn("flagged_at",  current_timestamp())

# Step 4: Summary of bad records by reason
print("=== Bad record breakdown ===")
bad_df.groupBy("fail_reason") \
      .count() \
      .orderBy(col("count").desc()) \
      .show()

# Step 5: Count unique users in valid data
unique_users = valid_df.select(
    countDistinct("user_id").alias("unique_users")
)
unique_users.show()

# Step 6: Write both zones
valid_df.write.mode("append").parquet("s3://bucket/processed/logs/")
bad_df.write.mode("append").json("s3://bucket/quarantine/logs/")

"I read with PERMISSIVE mode, tag each bad row with a fail_reason column using CASE WHEN, split into valid and bad DataFrames, write both to separate S3 zones, then run countDistinct on user_id on the clean DataFrame. I also group by fail_reason to show the breakdown of why records failed — that gives the team visibility into upstream data issues."
Q5 How do you get alerted when data quality fails or pipeline breaks ▼
The alerting stack on AWS
CloudWatchMonitors Glue job metrics — duration, records processed, DPU usage. Set alarms on job failure or high error rate.
SNSSimple Notification Service. Sends email or Slack message when CloudWatch alarm fires or when your code explicitly publishes a message.
Step FunctionsOrchestrates multi-step pipelines. Each step has retry logic and on-failure transitions to an alert step.
Pattern 1 — alert from within your Spark job
import boto3

def send_alert(subject, message, topic_arn):
    sns = boto3.client("sns", region_name="us-east-1")
    sns.publish(
        TopicArn=topic_arn,
        Subject=subject,
        Message=message
    )

# Use in pipeline
try:
    total = df.count()
    bad   = bad_df.count()

    # Alert if bad rate exceeds threshold
    if total > 0 and (bad / total) > 0.05:
        send_alert(
            subject="Data quality alert — orders pipeline",
            message=f"Bad record rate: {bad}/{total} = {100*bad/total:.1f}%\nJob: orders_pipeline\nDate: {date.today()}",
            topic_arn="arn:aws:sns:us-east-1:123456:pipeline-alerts"
        )

    # Alert if volume drops unexpectedly
    expected_min = 10_000
    if total < expected_min:
        send_alert(
            subject="Low volume alert — orders pipeline",
            message=f"Only {total} records received. Expected at least {expected_min}.",
            topic_arn="arn:aws:sns:us-east-1:123456:pipeline-alerts"
        )

except Exception as e:
    # Alert on any job failure
    send_alert(
        subject="FAILURE — orders pipeline",
        message=f"Pipeline failed with error: {str(e)}",
        topic_arn="arn:aws:sns:us-east-1:123456:pipeline-alerts"
    )
    raise   # re-raise so Glue marks job as failed
Pattern 2 — CloudWatch alarm on Glue job failure
# Terraform — create CloudWatch alarm for Glue job failure
resource "aws_cloudwatch_metric_alarm" "glue_failure" {
  alarm_name          = "orders-pipeline-failure"
  metric_name         = "glue.driver.aggregate.numFailedTasks"
  namespace           = "Glue"
  statistic           = "Sum"
  period              = 300
  evaluation_periods  = 1
  threshold           = 1
  comparison_operator = "GreaterThanOrEqualToThreshold"
  alarm_actions       = ["arn:aws:sns:us-east-1:123:pipeline-alerts"]
}
Say this in interview
"I have two layers of alerting. First, my pipeline code sends an SNS message if the bad record rate exceeds a threshold or if total volume drops unexpectedly — these are business logic alerts. Second, CloudWatch monitors the Glue job itself and fires an SNS alarm if the job fails or runs too long — these are infrastructure alerts. Both feed into the same SNS topic so the team gets one stream of notifications."
Two layers of alerting — business logic alerts from your code AND infrastructure alerts from CloudWatch. Mentioning both shows you think about monitoring at multiple levels.
