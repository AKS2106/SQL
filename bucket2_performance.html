Q1 Skewed data — what it is, how to detect it, how to fix it ▼
The core idea — understand this before the code
Spark splits data into partitions and runs each partition on a separate task. Skew means one partition has much more data than others — say 80% of rows have country = US. That one task runs for hours while all others finish in minutes. The whole job waits for that one slow task.
What skewed data looks like
PartitionRowsTime
country = US8,000,00045 mins
country = UK200,0002 mins
country = IN150,0001 min
→ Whole job waits 45 mins because of ONE partition
Step 1 — detect skew first
# Check how many rows are in each partition
from pyspark.sql.functions import spark_partition_id

df.groupBy(spark_partition_id().alias("partition")) \
  .count() \
  .orderBy("count", ascending=False) \
  .show()

# Check which key value is causing skew
df.groupBy("country") \
  .count() \
  .orderBy("count", ascending=False) \
  .show(10)
Fix 1 — Broadcast join (use when one table is small)
If one of your tables is small (under 50MB) just send it to every executor. No shuffle needed at all. This is the simplest and best fix.
from pyspark.sql.functions import broadcast

# Small lookup table — send to all executors
result = df_large.join(broadcast(df_small), "country")

# Or set threshold — Spark auto-broadcasts below this size
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50mb")
Fix 2 — Salting (use when both tables are large)
Add a random number (the "salt") to the skewed key. Now US_0, US_1, US_2 ... US_9 spread across 10 partitions instead of 1. The small table needs all 10 versions so you explode it.
from pyspark.sql.functions import rand, concat, lit, array, explode

SALT = 10  # split hot key into 10 buckets

# Step 1: add random salt to the large (skewed) table
df_large = df_large \
    .withColumn("salt",
        (rand() * SALT).cast("int").cast("string")) \
    .withColumn("salted_key",
        concat(col("country"), lit("_"), col("salt")))

# Step 2: explode small table so every salt version exists
df_small = df_small \
    .withColumn("salts",
        array([lit(str(i)) for i in range(SALT)])) \
    .withColumn("salt", explode("salts")) \
    .withColumn("salted_key",
        concat(col("country"), lit("_"), col("salt"))) \
    .drop("salts", "salt")

# Step 3: join on salted key — data now spreads evenly
result = df_large.join(df_small, "salted_key") \
                 .drop("salted_key", "salt")
Say this in interview
"Skew means one partition has far more data than others — like 80% of rows being country=US. That one task takes hours while others finish in minutes. I detect it by checking partition sizes. Fix: broadcast join if one table is small — simplest option. If both tables are large I use salting — add a random number to the join key so the hot key spreads across multiple partitions."
Salting trade-off: the small table grows by SALT factor and the code gets more complex. Always try broadcast join first — salting is the last resort.
Q2 Broadcast join vs Sort-Merge join — what's the difference ▼
The core idea — what "shuffle" means
A shuffle is when Spark moves data across the network between executors to line up matching keys for a join. It is the most expensive operation in Spark. The goal is to avoid or minimise shuffles.
Broadcast join
Small table sent to EVERY executor. Join happens locally — no network shuffle. Fastest join type. Use when one table is under ~50MB.
Sort-Merge join
Default for large-large joins. Both tables sorted and shuffled across network so matching keys end up on same executor. Expensive but handles any size.
How a broadcast join works step by step
Small table (50MB)
→
Driver collects it
→
Sends copy to every executor
→
Each executor joins locally
→
No shuffle needed
from pyspark.sql.functions import broadcast

# Explicit broadcast hint — always works
result = df_large.join(broadcast(df_small), "dept_id")

# Auto threshold — Spark broadcasts tables below this size
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50mb")

# ALWAYS verify which join type Spark actually used
result.explain()
# Look for: BroadcastHashJoin  ← broadcast worked
# vs:       SortMergeJoin      ← shuffle happened
Say this in interview
"Broadcast join sends the small table to every executor so the join happens locally with no shuffle — it's the fastest join type. Sort-merge join is the default for large tables — both sides get shuffled across the network so matching keys land on the same executor. I always check explain() to confirm which join Spark actually chose."
Always mention explain() — it shows you verify rather than assume. That one habit signals senior-level thinking to interviewers.
Q3 Cache vs Persist — the difference and when to use each ▼
The core idea — why caching exists
PySpark is lazy — every time you call an action (count, show, write) on a DataFrame it recomputes the whole lineage from scratch. If you use the same DataFrame 3 times that computation runs 3 times. Cache stores the result so it only computes once.
cache() vs persist() — what is the actual difference
cache()Shorthand for persist with MEMORY_AND_DISK level. Quick to write. Good default choice.
persist(level)Same as cache but you choose the storage level. Use when you need control over where data is stored.
Storage levels — which to pick
MEMORY_ONLYFastest. Stored in RAM only. If not enough RAM — data is recomputed (not spilled). Use for small DataFrames that fit in memory.
MEMORY_AND_DISKDefault via cache(). Stores in RAM, spills to disk if RAM is full. Safer choice for production.
DISK_ONLYStored on disk only. Slow but useful for very large DataFrames where you want to avoid recomputation but can't fit in RAM.
from pyspark import StorageLevel

# cache() — shorthand, MEMORY_AND_DISK
df.cache()

# persist() — choose your level
df.persist(StorageLevel.MEMORY_ONLY)
df.persist(StorageLevel.MEMORY_AND_DISK)   # same as cache()
df.persist(StorageLevel.DISK_ONLY)

# ---- Real usage pattern ----
# Step 1: cache the DataFrame
df_active = df.filter(col("status") == "active").cache()

# Step 2: trigger caching with an action
# cache() is LAZY — nothing is stored until an action runs
count = df_active.count()           # this triggers the cache

# Step 3: reuse — Spark reads from cache, not recomputes
agg1 = df_active.groupBy("dept").count()
agg2 = df_active.agg(avg("salary"))

# Step 4: ALWAYS unpersist when done — free memory
df_active.unpersist()
Say this in interview
"cache() is shorthand for persist with MEMORY_AND_DISK level. persist() gives control to choose the storage level. I use cache when a DataFrame is used multiple times in the same job — it prevents Spark recomputing from scratch each time. Important: cache is lazy so I always call .count() right after to force it into memory. And I always unpersist when done to free up memory."
The most common mistake: people cache() but never unpersist(). The cached data stays in memory for the rest of the job eating RAM. Always unpersist when you're done with that DataFrame.
Q4 Partitioning vs Bucketing — the difference ▼
The one-line difference
Partitioning = organise files into folders by column value. Speeds up reads by skipping folders. Bucketing = distribute rows into fixed buckets by hash. Speeds up joins and aggregations by eliminating shuffle.
Partitioning
Creates folder structure in S3.
s3://bucket/year=2024/month=01/

Spark skips folders that don't match your filter — called partition pruning.

Use for: date, region, status — low cardinality columns.
Bucketing
Hash distributes rows into N fixed buckets.
All rows with same key go to same bucket.

When joining two bucketed tables on the same key — no shuffle needed.

Use for: user_id, order_id — high cardinality join keys.
# --- PARTITIONING ---
# Write — creates folder structure
df.write \
  .partitionBy("year", "month") \
  .parquet("s3://bucket/sales/")
# Result: s3://bucket/sales/year=2024/month=01/file.parquet
#         s3://bucket/sales/year=2024/month=02/file.parquet

# Read — Spark skips non-matching folders (partition pruning)
spark.read.parquet("s3://bucket/sales/") \
     .filter(col("year") == 2024)
# Spark reads ONLY year=2024 folder — ignores everything else


# --- BUCKETING ---
# Write — hash distributes rows into 16 buckets by customer_id
df.write \
  .bucketBy(16, "customer_id") \
  .sortBy("customer_id") \
  .saveAsTable("orders_bucketed")

# Join two bucketed tables — NO shuffle required
orders  = spark.read.table("orders_bucketed")
customers = spark.read.table("customers_bucketed")
result = orders.join(customers, "customer_id")
# Both tables pre-bucketed on customer_id = matching rows
# are already on the same executor
Say this in interview
"Partitioning creates a folder structure by column value — great for filtering because Spark skips non-matching folders, which is called partition pruning. Bucketing pre-distributes rows into fixed hash buckets — when two tables are bucketed on the same join key there is no shuffle at join time. Partitioning is for read filtering, bucketing is for join optimisation."
Partitioning column rule: low cardinality only — date, region, status. If you partition by user_id you get millions of tiny folders — that is worse than no partitioning at all.
Q5 Efficient joins for large data — which strategy when ▼
The decision tree — ask these questions in order
Is one table small?Yes → broadcast join. Fastest. No shuffle. Done.
Is data skewed?Yes → salting (Q1). Distribute hot keys across partitions.
Repeated joins?Yes → bucketing. Pre-bucket both tables once, all future joins on that key are shuffle-free.
None of above?Sort-Merge join (Spark default). Make sure both tables are well-partitioned to minimise shuffle size.
# Strategy 1: broadcast (small table)
result = large_df.join(broadcast(small_df), "key")

# Strategy 2: salting (skewed large-large join) — see Q1

# Strategy 3: bucketing (repeated joins on same key)
orders.write.bucketBy(32,"customer_id").saveAsTable("orders_b")
customers.write.bucketBy(32,"customer_id").saveAsTable("cust_b")
# All future joins on customer_id = no shuffle

# Strategy 4: repartition before join (one-time optimisation)
# If you know both tables will be joined on user_id,
# repartition both on user_id first to co-locate matching rows
df_a = df_a.repartition(200, "user_id")
df_b = df_b.repartition(200, "user_id")
result = df_a.join(df_b, "user_id")

# Always verify — check explain()
result.explain()
# BroadcastHashJoin = best
# SortMergeJoin     = acceptable for large-large
# CartesianProduct  = never — means join condition is missing
Say this in interview
"My first question is always — is one table small enough to broadcast? If yes, broadcast join, no shuffle needed. If both tables are large and data is skewed I use salting. If the same join key is used repeatedly across the job I use bucketing so shuffle only happens once at write time. I always check explain() to confirm what Spark actually chose."
CartesianProduct in explain() means you wrote a join without a condition — every row joins every other row. On a million-row table that is a trillion rows. Always check explain() before running a join on large data.
