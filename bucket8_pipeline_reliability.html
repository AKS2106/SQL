Q1 How do you handle a job that fails halfway through ▼
The core idea — idempotency
The most important word in this answer is idempotent. A pipeline is idempotent when running it twice gives exactly the same result as running it once. If your job fails halfway and you re-run it — the output is correct, no duplicates, no missing data.
Non-idempotent — dangerous
Job fails halfway. You re-run it. First half of data is written twice. Output has duplicates. Data is wrong.
Idempotent — safe
Job fails halfway. You re-run it. Write mode is overwrite. Same output as if it ran once. Always safe to retry.
Three techniques to make a pipeline idempotent
# Technique 1: partition overwrite — safest for batch jobs
# Writing same partition twice = same result every time
df.write \
  .mode("overwrite") \
  .partitionBy("event_date") \
  .parquet("s3://bucket/processed/")

# Technique 2: Delta Lake MERGE — handles duplicates automatically
# Re-running MERGE with same data = same result (update not insert)
target.alias("t").merge(
    updates_df.alias("s"),
    "t.order_id = s.order_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

# Technique 3: write to temp location, then atomic rename
# Either fully committed or not — no partial writes visible
TEMP_PATH = "s3://bucket/temp/orders_20240115/"
FINAL_PATH = "s3://bucket/processed/orders/dt=2024-01-15/"

df.write.mode("overwrite").parquet(TEMP_PATH)  # write to temp
# If write succeeded, rename to final (atomic in most storage)
import boto3
s3 = boto3.client("s3")
# copy from temp to final, delete temp
Handle corrupted input records without failing the whole job
try:
    df = spark.read \
        .option("mode", "PERMISSIVE") \
        .option("columnNameOfCorruptRecord", "_corrupt") \
        .schema(schema) \
        .json("s3://bucket/raw/")

    valid_df = df.filter(col("_corrupt").isNull()).drop("_corrupt")
    bad_df   = df.filter(col("_corrupt").isNotNull())

    # Write valid records — job continues even if some records are bad
    valid_df.write.mode("overwrite") \
        .partitionBy("dt") \
        .parquet("s3://bucket/processed/")

    # Preserve bad records for investigation
    if bad_df.count() > 0:
        bad_df.write.mode("append").json("s3://bucket/quarantine/")

except Exception as e:
    # Notify team — job failed completely
    boto3.client("sns").publish(
        TopicArn="arn:aws:sns:us-east-1:123:alerts",
        Subject="Pipeline failed",
        Message=str(e)
    )
    raise  # re-raise so Glue marks job as FAILED
Say this in interview
"The key is idempotency — designing the pipeline so re-running it gives the same result as running it once. I achieve this with partition overwrite mode so writing the same partition twice does not create duplicates, or with Delta Lake MERGE which naturally handles re-runs. For corrupted input I use PERMISSIVE mode — bad records go to quarantine, valid records continue, the job does not fail because of a few bad rows."
Always say idempotent when answering this question. Then explain how you achieve it. That one word shows you think about failures proactively not reactively.
Q2 How do you get alerted when a pipeline fails ▼
Two layers of alerting — always mention both
Layer 1 — code alertsYour pipeline code sends SNS when business logic fails — bad record rate too high, row count too low, watermark not advancing.
Layer 2 — infra alertsCloudWatch monitors the Glue job itself — job failed, job ran too long, too many errors. Fires SNS alarm automatically.
import boto3
from datetime import date

SNS_TOPIC = "arn:aws:sns:us-east-1:123456:pipeline-alerts"
sns = boto3.client("sns")

# --- Layer 1: business logic alerts from your code ---
def check_and_alert(df, job_name):
    total     = df.count()
    bad_count = df.filter(col("_corrupt").isNotNull()).count()

    # Alert if bad record rate too high
    if total > 0 and bad_count / total > 0.05:
        sns.publish(
            TopicArn=SNS_TOPIC,
            Subject=f"Data quality alert — {job_name}",
            Message=(
                f"Job: {job_name}\n"
                f"Date: {date.today()}\n"
                f"Bad records: {bad_count}/{total} "
                f"({100*bad_count/total:.1f}%)\n"
                f"Check quarantine: s3://bucket/quarantine/"
            )
        )

    # Alert if row count drops more than 20% vs yesterday
    # (compare to watermark stored in DynamoDB)
    expected_min = 50_000
    if total < expected_min:
        sns.publish(
            TopicArn=SNS_TOPIC,
            Subject=f"Low volume alert — {job_name}",
            Message=f"Only {total} rows. Expected at least {expected_min}."
        )

# --- Layer 2: infra alert — wrap entire job in try/except ---
def run_pipeline():
    try:
        df = spark.read.parquet("s3://bucket/raw/")
        check_and_alert(df, "orders_pipeline")
        # ... rest of pipeline ...
        df.write.mode("overwrite").parquet("s3://bucket/processed/")
    except Exception as e:
        sns.publish(
            TopicArn=SNS_TOPIC,
            Subject="FAILURE — orders_pipeline",
            Message=f"Error: {str(e)}\nDate: {date.today()}"
        )
        raise  # re-raise — Glue marks job as FAILED, CloudWatch alarm fires
CloudWatch alarm — automatic infra-level alerting
# Terraform — CloudWatch alarm fires SNS when Glue job fails

  alarm_name          = "orders-pipeline-failure"

  statistic           = "Sum"

  evaluation_periods  = 1

  comparison_operator = "GreaterThanOrEqualToThreshold"
  alarm_actions       = [aws_sns_topic.alerts.arn]
  ok_actions          = [aws_sns_topic.alerts.arn]
}
Say this in interview
"I have two layers. First, my pipeline code sends SNS alerts for business logic failures — bad record rate over 5%, row count below expected threshold, any exception in the job. Second, CloudWatch monitors the Glue job itself and fires a separate SNS alarm if the job fails or runs too long. Both feed into the same SNS topic so the team gets one stream of notifications. I always re-raise after sending the code alert so CloudWatch also sees the failure — otherwise CloudWatch thinks the job succeeded."
Q3 Kinesis — what it is and how it fits in a real-time pipeline ▼
What Kinesis Data Streams is
Kinesis is AWS's managed real-time data streaming service — like Kafka but fully managed. Data flows through shards. Each shard handles 1MB/s write and 2MB/s read. You scale by adding shards.
ShardUnit of capacity. 1MB/s in, 2MB/s out. Add shards to scale throughput. Each shard is independent — parallelism comes from multiple shards.
Partition keyDetermines which shard a record goes to. Use user_id for even distribution. Using a single key sends everything to one shard — skew.
RetentionDefault 24 hours, max 7 days. Events are replayable within the retention window — if processing fails, replay from the beginning.
vs KafkaKinesis is fully managed — no cluster to run. Kafka gives more control and is open source. On AWS, use Kinesis. On-prem or multi-cloud, use Kafka.
# Producer — write events to Kinesis (from app, IoT device, or Lambda)
import boto3, json

kinesis = boto3.client("kinesis", region_name="us-east-1")

def send_event(user_id, event_type, amount):
    kinesis.put_record(
        StreamName="user-events",
        Data=json.dumps({
            "user_id":    user_id,
            "event_type": event_type,
            "amount":     amount,
            "ts":         "2024-01-15T10:30:00Z"
        }),
        PartitionKey=user_id   # routes to shard — use user_id for even spread
    )

# Consumer — Spark Structured Streaming reads from Kinesis
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import *

schema = StructType([
    StructField("user_id",    StringType()),
    StructField("event_type", StringType()),
    StructField("amount",     DoubleType()),
    StructField("ts",         StringType())
])

df_stream = spark.readStream \
    .format("kinesis") \
    .option("streamName",       "user-events") \
    .option("region",           "us-east-1") \
    .option("initialPosition",  "LATEST") \
    .load()

# Parse JSON payload from Kinesis record
df_parsed = df_stream \
    .withColumn("data",
        from_json(col("data").cast("string"), schema)) \
    .select("data.*")

# Apply watermark and window — with late data handling
windowed = df_parsed \
    .withWatermark("ts", "5 minutes") \
    .groupBy(
        window(col("ts").cast("timestamp"), "1 hour"),
        "event_type"
    ).agg(
        count("*").alias("event_count"),
        sum("amount").alias("total_amount")
    )

# Write to S3 every 30 seconds
query = windowed.writeStream \
    .outputMode("update") \
    .format("parquet") \
    .option("path",            "s3://bucket/streaming/hourly_events/") \
    .option("checkpointLocation","s3://bucket/checkpoints/kinesis/") \
    .trigger(processingTime="30 seconds") \
    .start()

query.awaitTermination()
Say this in interview
"Kinesis is AWS's managed streaming service — like Kafka but serverless. Data flows through shards, each handling 1MB/s write. Producers write events with a partition key that routes to a shard — I use user_id for even distribution. Consumers are Spark Structured Streaming jobs that read from shards, apply watermarking for late data, and write windowed aggregations to S3. I always set a checkpoint location so if the streaming job crashes it resumes from where it left off, not the beginning."
Checkpoint location is the streaming equivalent of idempotency. Without it, a streaming job restart replays all events from the start and creates duplicates. Always mention it when talking about Kinesis or Spark Streaming.
Q4 Deploying Spark jobs with CI/CD ▼
The three parts of a DE CI/CD pipeline
1. TestUnit tests on PySpark transformations using pytest + local Spark. Runs on every pull request.
2. PackageUpload updated script to S3. Update Glue job definition to point to new script.
3. InfrastructureGlue job definitions, IAM roles, S3 buckets managed in Terraform — version controlled alongside code.
# --- Unit test: test PySpark logic without a real cluster ---
# tests/test_transforms.py

import pytest
from pyspark.sql import SparkSession
from pyspark.sql.functions import col
from src.transform import add_readmission_flag, dept_summary

@pytest.fixture(scope="session")
def spark():
    return SparkSession.builder \
        .master("local[1]") \
        .appName("unit_tests") \
        .getOrCreate()

def test_readmission_flag(spark):
    data = [
        ("P1", "2024-01-01", "2024-01-05"),   # first admission
        ("P1", "2024-01-20", "2024-01-25"),   # 15 days later = readmit
        ("P2", "2024-01-01", "2024-01-10"),   # only one admission
    ]
    df = spark.createDataFrame(
        data, ["patient_id","admission_date","discharge_date"]
    )
    result = add_readmission_flag(df)
    # P1 second admission should be flagged as readmission
    readmits = result.filter(col("is_readmission") == 1).count()
    assert readmits == 1, f"Expected 1 readmission, got {readmits}"

def test_no_negative_los(spark):
    # Length of stay should never be negative
    data = [("P1","2024-01-01","2024-01-05")]
    df = spark.createDataFrame(
        data, ["patient_id","admission_date","discharge_date"]
    )
    result = add_readmission_flag(df)
    neg_los = result.filter(col("los_days") < 0).count()
    assert neg_los == 0, "Negative length of stay detected"

Q5 Design an end-to-end pipeline — how to answer in 2 minutes ▼
The 5-layer structure — use this for any pipeline question
When an interviewer says "design a pipeline for X" — do not start drawing boxes immediately. Say this first: "I think about it in 5 layers — ingestion, storage, processing, serving, and monitoring. Let me walk through each."
1. IngestionHow does data arrive? Batch (S3 file drop, API pull, database export) or streaming (Kinesis, Kafka)? What format? What volume per day?
2. StorageRaw zone: S3, original format, never delete. Processed zone: S3, Parquet or Delta, partitioned by date. Serving layer: Redshift, Athena, or DynamoDB.
3. ProcessingGlue + PySpark for batch ETL. Spark Structured Streaming for real-time. Airflow or MWAA for orchestration and dependency management.
4. ServingAthena for ad-hoc SQL queries on S3. Redshift for dashboards with frequent queries. DynamoDB for millisecond lookups from applications.
5. MonitoringCloudWatch metrics on Glue jobs. SNS alerts for failures and data quality issues. Row count reconciliation between source and destination.
# --- Healthcare batch pipeline — concrete example ---

# Layer 1: Ingestion
# Hospital EHR exports patient records to S3 daily at 2am
# Format: CSV, ~500k rows/day
# Lambda triggered by S3 PUT event → starts Glue job

# Layer 2: Storage
# s3://hospital-datalake/raw/admissions/dt=2024-01-15/     ← raw, never touched
# s3://hospital-datalake/processed/admissions/dt=2024-01-15/ ← Parquet, partitioned
# s3://hospital-datalake/delta/patient_summary/            ← Delta for SCD Type 2

# Layer 3: Processing — Glue ETL job
def run_admissions_pipeline(spark, glueContext, input_path, run_date):
    # Read
    dyf = glueContext.create_dynamic_frame.from_catalog(
        database="hospital_db", table_name="raw_admissions",
        push_down_predicate=f"dt = '{run_date}'"
    )
    df = dyf.toDF()

    # Quality checks
    bad_df  = df.filter(col("patient_id").isNull())
    good_df = df.filter(col("patient_id").isNotNull())

    # Transform — add readmission flag
    from pyspark.sql.window import Window
    w = Window.partitionBy("patient_id").orderBy("admission_date")
    good_df = good_df \
        .withColumn("prev_discharge", lag("discharge_date").over(w)) \
        .withColumn("is_readmission",
            when(datediff("admission_date","prev_discharge") <= 30, 1)
            .otherwise(0))

    # Write
    good_df.write.mode("overwrite").partitionBy("dt") \
        .parquet("s3://hospital-datalake/processed/admissions/")
    bad_df.write.mode("append") \
        .json("s3://hospital-datalake/quarantine/admissions/")

# Layer 4: Serving
# Athena queries on processed/ for ad-hoc analysis
# QuickSight dashboard connected to Athena for clinical ops team

# Layer 5: Monitoring
# CloudWatch alarm on Glue job failure → SNS → email team
# Row count check: source CSV rows == processed Parquet rows
Say this in interview
"I design pipelines in 5 layers. Ingestion: how and when data arrives — batch file drop or streaming events. Storage: raw zone keeps original data forever, processed zone has clean partitioned Parquet or Delta. Processing: Glue PySpark for batch, Spark Streaming for real-time, Airflow for orchestration. Serving: Athena for ad-hoc, Redshift for dashboards, DynamoDB for low-latency app lookups. Monitoring: CloudWatch alarms plus SNS alerts from code for business logic failures. Every layer has a clear purpose."
This structure works for any domain — healthcare, auto, finance, retail. The 5 layers never change. Only the specific technologies and data volumes change per scenario.
Q6 How do you handle pipeline failures and recover — the complete playbook ▼
Failure types — each needs a different response
Infrastructure failureGlue job crashes, EMR cluster dies, Lambda times out. Fix: retry with exponential backoff via Step Functions or Glue retry config.
Bad input dataCorrupt file, schema mismatch, unexpected NULL. Fix: quarantine bad records, continue with valid data, alert team.
Logic errorBug in transform produced wrong output. Fix: restore from Delta time travel or reprocess from raw zone.
Upstream delaySource data arrived 3 hours late. Fix: pipeline waits with dependency check or handles late data with reprocessing window.
# --- Retry config in Glue ---
# Set in Glue job config:
# Max retries: 3
# Retry delay: 2 minutes
# This handles transient failures (network blip, S3 throttle)

# --- Step Functions with exponential backoff ---
{
  "Type": "Task",
  "Resource": "arn:aws:states:...:glue:startJobRun.sync",
  "Parameters": {"JobName": "orders_pipeline"},
  "Retry": [{
    "ErrorEquals": ["States.ALL"],
    "IntervalSeconds": 60,       # wait 60s before first retry
    "MaxAttempts": 3,            # retry up to 3 times
    "BackoffRate": 2.0           # double wait time each retry
  }],
  "Catch": [{
    "ErrorEquals": ["States.ALL"],
    "Next": "NotifyFailure"      # after all retries fail → alert
  }],
  "Next": "NextStep"
}

# --- Recovery from logic error using Delta time travel ---
from delta.tables import DeltaTable

# Check what versions exist
spark.sql("""
    DESCRIBE HISTORY delta.`s3://bucket/delta/orders/`
""").show()

# Restore to version before the bad job ran
dt = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")
dt.restoreToVersion(12)   # version 12 was last known good state
print("Table restored to version 12")
The recovery playbook — say this structure in interviews
Step 1 — DetectCloudWatch alarm fires. SNS alert goes to team. On-call engineer gets paged within 5 minutes of failure.
Step 2 — AssessCheck Glue job logs in CloudWatch. Is it infra failure (retry) or data issue (quarantine and reprocess)?
Step 3 — FixInfra failure: retry job — idempotent writes mean safe to re-run. Data issue: fix quarantine records, reprocess. Logic bug: restoreToVersion in Delta, fix code, redeploy.
Step 4 — VerifyRow count reconciliation — source count matches destination count. Data quality checks pass. Downstream dashboards show correct data.
Step 5 — PreventAdd the failure scenario to unit tests. Add a data quality check that would have caught it. Document the incident.
Say this in interview
"My pipeline handles failures at three levels. Transient infra failures are handled automatically with Step Functions retry and exponential backoff — no human needed. Bad input data goes to quarantine with a fail_reason tag, valid data continues, and SNS alerts the team. Logic errors are recovered using Delta Lake restoreToVersion — one line of code to roll back to the last good state. After any failure I add the scenario to unit tests and data quality checks so it cannot silently happen again."
The phrase "Step 5 — Prevent" is what makes this a senior answer. Juniors fix the immediate problem. Seniors fix it and make sure it never happens silently again.
