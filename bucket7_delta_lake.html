Bucket 7 — Delta Lake

6 questions. Delta Lake is one of the hottest topics in DE interviews right now. The core idea: Delta Lake is Parquet files plus a transaction log. That log is what gives you ACID, time travel, MERGE, and schema evolution. Understand that and every question unlocks.
Q1 Delta Lake vs Parquet — what is the difference and when to use each ▼
The core idea — what Delta Lake actually is
Delta Lake is NOT a different file format. It is Parquet files plus a transaction log folder called _delta_log/. That log is a JSON file that records every change ever made to the table. The log is what gives Delta all its superpowers.

When to choose each
Use Parquet whenPipeline is append-only. No corrections needed. Cost-sensitive. Simple batch jobs. No need for history.
Use Delta whenNeed to update or delete records. Late-arriving corrections. SCD Type 2. Schema changes frequently. Need audit trail.
# Writing Delta — same as Parquet but format="delta"
df.write.format("delta") \
    .mode("overwrite") \
    .partitionBy("year","month") \
    .save("s3://bucket/delta/orders/")

# Reading Delta
df = spark.read.format("delta") \
    .load("s3://bucket/delta/orders/")

# Or register as table and use SQL
spark.sql("CREATE TABLE orders USING DELTA LOCATION 's3://bucket/delta/orders/'")
spark.sql("SELECT * FROM orders WHERE year = 2024")
Say this in interview
"Delta Lake is not a new file format — it is Parquet files plus a transaction log folder called _delta_log. That log records every change and is what gives Delta ACID transactions, MERGE for upserts, time travel to query past versions, and managed schema evolution. I use plain Parquet for simple append-only pipelines. I use Delta when I need to update records, handle late-arriving corrections, or implement SCD Type 2."
Always say "_delta_log folder" when explaining Delta Lake. It shows you understand the implementation not just the marketing. That one detail signals real knowledge.
Q2 Delta Lake MERGE — the upsert pattern ▼
What MERGE does — in plain English
MERGE looks at each incoming record and asks: does this record already exist in my table? If yes — update it. If no — insert it. The whole operation is atomic — either all changes succeed or none of them do.
Incoming record
→
Match on key?
→
Yes → UPDATE
|
No → INSERT
Why this matters — the problems MERGE solves
Late correctionsOrder arrives late with updated amount. MERGE updates the existing record — no full rewrite needed.
DeduplicationSame order arrives twice. MERGE updates instead of inserting a second copy — no duplicates.
SCD Type 2Customer changes address. MERGE closes old record and inserts new one with history preserved.
from delta.tables import DeltaTable

# Target: existing Delta table
target = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")

# Source: new/updated records coming in
updates_df = spark.read.parquet("s3://bucket/raw/order_updates/")

# MERGE — upsert pattern
target.alias("t").merge(
    updates_df.alias("s"),
    "t.order_id = s.order_id"     # match condition — join key
).whenMatchedUpdate(set={         # if record EXISTS → update these cols
    "status":     "s.status",
    "amount":     "s.amount",
    "updated_at": "s.updated_at"
}).whenNotMatchedInsert(values={  # if record NEW → insert all cols
    "order_id":   "s.order_id",
    "customer_id":"s.customer_id",
    "status":     "s.status",
    "amount":     "s.amount",
    "created_at": "s.created_at",
    "updated_at": "s.updated_at"
}).execute()

# Shorthand — update ALL columns when matched, insert ALL when new
target.alias("t").merge(
    updates_df.alias("s"),
    "t.order_id = s.order_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()
MERGE with delete — handle soft deletes
# If source marks records as deleted, remove them from target
target.alias("t").merge(
    updates_df.alias("s"),
    "t.order_id = s.order_id"
).whenMatchedDelete(
    condition="s.is_deleted = true"   # delete if flagged
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()
Say this in interview
"MERGE is Delta Lake's upsert — if a record matches on the join key I update it, if not I insert it. The whole operation is atomic — either everything succeeds or everything rolls back. I use it for late-arriving corrections so I never need a full table rewrite, for deduplication so duplicates naturally get updated instead of inserted twice, and for SCD Type 2 to manage slowly changing dimensions."
The word atomic is key here. Parquet has no atomicity — if your write crashes halfway, you have partial data. Delta's transaction log means the write either fully commits or fully rolls back. That is the ACID guarantee.
Q3 Time travel — query past versions of your data ▼
What time travel is
Every change to a Delta table creates a new version in the transaction log. Delta keeps the old Parquet files around until you explicitly clean them up. This means you can query your table as it looked at any past point in time.
Why this is useful in production
Audit requirement"What did the orders table look like on January 1st?" — regulators ask this.
Debugging"A bad job ran yesterday and corrupted data — show me what it looked like before." Then restore it.
Reproducibility"Re-train the ML model on data as of 3 months ago." Read exactly that snapshot.
# Time travel by version number
df_v0 = spark.read.format("delta") \
    .option("versionAsOf", 0) \
    .load("s3://bucket/delta/orders/")

# Time travel by timestamp
df_yesterday = spark.read.format("delta") \
    .option("timestampAsOf", "2024-01-14 00:00:00") \
    .load("s3://bucket/delta/orders/")

# See all versions — what changed and when
spark.sql("""
    DESCRIBE HISTORY delta.`s3://bucket/delta/orders/`
""").select(
    "version","timestamp","operation","operationParameters"
).show(10, truncate=False)

# Restore table to a previous version (undo a bad job)
from delta.tables import DeltaTable
dt = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")
dt.restoreToVersion(5)         # restore to version 5
# or
dt.restoreToTimestamp("2024-01-13 09:00:00")  # restore to timestamp
How long versions are kept — VACUUM
# By default Delta keeps 30 days of history
# Old files not cleaned up automatically — VACUUM does it

# Run VACUUM to delete files older than retention period
spark.sql("""
    VACUUM delta.`s3://bucket/delta/orders/`
    RETAIN 168 HOURS   -- keep 7 days of history
""")

# WARNING: after VACUUM you CANNOT time travel beyond retention period
# Default retention is 7 days minimum — do not go below this
Say this in interview
"Delta Lake keeps every version of the table in its transaction log. I can query any past version by timestamp or version number — useful for audits, debugging bad pipeline runs, or ML model reproducibility. DESCRIBE HISTORY shows me what changed and when. If a bad job corrupts data I use restoreToVersion to roll back instantly. I run VACUUM periodically to clean up old files while keeping a sensible retention window like 7 days."
restoreToVersion is the production answer to "how do you recover from a bad pipeline run." Without Delta you would need to reprocess from raw data. With Delta it is one line of code.
Q4 Schema evolution and schema enforcement in Delta Lake ▼
Two modes — know both
Schema enforcement (default)
Delta rejects writes that do not match the existing schema. Protects data integrity. Good default for production — catches unexpected upstream changes immediately.
Schema evolution
Delta automatically adds new columns when mergeSchema=true. Safe for planned schema changes. Use when you know upstream is adding columns.
# Schema enforcement — default behaviour
df_new = spark.read.parquet("s3://bucket/raw/")
df_new.write.format("delta").mode("append") \
    .save("s3://bucket/delta/orders/")
# If df_new has a new column → AnalysisException: schema mismatch
# This is GOOD — it catches unexpected upstream changes

# Schema evolution — add new columns automatically
df_new.write.format("delta") \
    .option("mergeSchema", "true") \
    .mode("append") \
    .save("s3://bucket/delta/orders/")
# New columns added to table schema automatically
# Old rows get NULL for the new column

# Check current schema
dt = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")
dt.toDF().printSchema()

# Full schema change history — when each column was added
spark.sql("""
    DESCRIBE HISTORY delta.`s3://bucket/delta/orders/`
""").show()

# Add column with ALTER TABLE — explicit and controlled
spark.sql("""
    ALTER TABLE delta.`s3://bucket/delta/orders/`
    ADD COLUMNS (loyalty_tier STRING COMMENT 'Added 2024-01-15')
""")
What if column TYPES change — the dangerous case
# Delta rejects type changes by default — this is intentional
# amount was DoubleType, now source sends StringType — rejected

# Fix: cast incoming data to match existing schema
df_fixed = df_new.withColumn(
    "amount", col("amount").cast(DoubleType())
)
df_fixed.write.format("delta").mode("append") \
    .save("s3://bucket/delta/orders/")
Say this in interview
"Delta has two schema modes. Enforcement is the default — it rejects writes that don't match the schema. This protects data integrity and catches unexpected upstream changes immediately, which I want in production. For planned changes like a new column being added, I enable mergeSchema=true. Type changes are always explicit — I cast incoming data to match. The full schema change history is in DESCRIBE HISTORY so I always know what changed and when."
Schema enforcement catching unexpected changes is a feature not a limitation. Say it that way in interviews — "I keep enforcement on by default because I want to know immediately when upstream changes schema unexpectedly."
Q5 OPTIMIZE and ZORDER — fixing small files and speeding up queries ▼
Two problems — one command solves both
Small filesMany small Parquet files build up over time from streaming writes or frequent appends. OPTIMIZE compacts them into larger files.
Slow queriesQueries on high-cardinality columns like customer_id scan many files. ZORDER co-locates related data so fewer files are read.
# OPTIMIZE — compact small files into larger ones (target ~1GB each)
spark.sql("""
    OPTIMIZE delta.`s3://bucket/delta/orders/`
""")

# OPTIMIZE with ZORDER — compact AND co-locate by query column
# Best for columns you frequently filter or join on
spark.sql("""
    OPTIMIZE delta.`s3://bucket/delta/orders/`
    ZORDER BY (customer_id)
""")

# Multiple ZORDER columns — most selective first
spark.sql("""
    OPTIMIZE delta.`s3://bucket/delta/orders/`
    ZORDER BY (customer_id, order_date)
""")
When to run OPTIMIZE
After streaming writesStreaming creates many tiny files. Schedule OPTIMIZE every few hours as a maintenance job.
After MERGEMERGE rewrites files and can fragment them. Run OPTIMIZE after large MERGE operations.
Scheduled maintenanceAdd to your pipeline as a final step or run as a separate daily job.
# Auto-optimize — Delta automatically compacts on write
spark.conf.set(
    "spark.databricks.delta.autoOptimize.optimizeWrite","true"
)
spark.conf.set(
    "spark.databricks.delta.autoOptimize.autoCompact","true"
)

# After OPTIMIZE — VACUUM removes old files no longer needed
spark.sql("""
    VACUUM delta.`s3://bucket/delta/orders/`
    RETAIN 168 HOURS
""")
Say this in interview
"OPTIMIZE compacts small files into larger ones — same problem as coalesce but Delta manages it automatically. ZORDER goes a step further — it physically co-locates rows with similar values for a column so queries filtering on that column scan far fewer files. I run OPTIMIZE after streaming writes which create many tiny files, and after large MERGE operations. Then VACUUM to clean up the old files."
ZORDER is most effective on high-cardinality columns you frequently filter on — customer_id, product_id, user_id. Do not ZORDER on low-cardinality columns like status or year — partitioning handles those better.
Q6 SCD Type 2 with Delta Lake MERGE — the complete pattern ▼
What SCD Type 2 is
Slowly Changing Dimension Type 2 keeps full history when a dimension value changes. When a customer moves from London to Manchester — you do not overwrite London. You close the London record with an end date and insert a new Manchester record. Both rows exist so you can answer: "what city was this customer in when they placed order #123?"
customer_idcitystart_dateend_dateis_current
C001London2023-01-012024-01-10false
C001Manchester2024-01-10nulltrue
from delta.tables import DeltaTable
from pyspark.sql.functions import current_date, lit

target = DeltaTable.forPath(spark, "s3://bucket/delta/customers/")

# Incoming updates — customers whose data changed
updates_df = spark.read.parquet("s3://bucket/raw/customer_updates/")

# Step 1: Close existing current records that are changing
target.alias("t").merge(
    updates_df.alias("s"),
    """t.customer_id = s.customer_id
       AND t.is_current = true
       AND t.city != s.city"""       # only if city actually changed
).whenMatchedUpdate(set={
    "end_date":   "current_date()",  # close old record today
    "is_current": "false"            # mark as historical
}).execute()

# Step 2: Insert new current records for changed customers
new_records = updates_df \
    .withColumn("start_date", current_date()) \
    .withColumn("end_date",   lit(None).cast("date")) \
    .withColumn("is_current", lit(True))

new_records.write.format("delta") \
    .mode("append") \
    .save("s3://bucket/delta/customers/")

# Query: what city was customer C001 in on 2023-06-15?
spark.sql("""
    SELECT city FROM delta.`s3://bucket/delta/customers/`
    WHERE customer_id = 'C001'
    AND   start_date <= '2023-06-15'
    AND   (end_date > '2023-06-15' OR end_date IS NULL)
""")
Say this in interview
"SCD Type 2 keeps full history when dimension data changes. With Delta Lake I use two steps: first a MERGE to close the existing current record by setting end_date to today and is_current to false — only for records where the value actually changed. Then append the new record with today as start_date, null end_date, and is_current true. Delta's ACID guarantees both steps are atomic so I never end up with two current records for the same customer."
SCD Type 2 with Delta is a complete answer that combines MERGE, ACID, and data modelling knowledge in one question. If you can explain this clearly you are demonstrating 5 years of thinking in one answer.
