Bucket 5 — AWS Glue

6 questions. Glue is AWS's managed ETL service built on top of Spark. These questions test whether you understand how Glue adds to Spark — and when to use Glue-specific features vs plain PySpark.
Q1 Glue DynamicFrame vs Spark DataFrame — what is the difference ▼
The core idea
DynamicFrame is Glue's own data structure — a wrapper around DataFrame designed to handle messy real-world data where schemas are inconsistent. DataFrame is standard PySpark — faster and has the full API.
FeatureDynamicFrameDataFrame
Mixed types in columnHandles it — creates choice typeFails or casts to string
Missing columnsHandles gracefullyFails if schema mismatch
PerformanceSlowerFaster
Window functionsNot availableFull support
Glue Catalog I/ONative supportVia spark.read.table()
When to useIngestion onlyAll transformations
The golden rule — convert early
Use DynamicFrame only to read from Glue Catalog and write the final output. Convert to DataFrame immediately for everything in between.
from awsglue.dynamicframe import DynamicFrame
from pyspark.sql.functions import col, year

# Step 1: Read from Glue Catalog as DynamicFrame
dyf = glueContext.create_dynamic_frame.from_catalog(
    database="my_db",
    table_name="raw_orders",
    transformation_ctx="raw_orders"
)

# Step 2: Convert to DataFrame IMMEDIATELY
# Do ALL transformations as DataFrame
df = dyf.toDF()

df_clean = df \
    .filter(col("status") == "active") \
    .withColumn("order_year", year("order_date")) \
    .dropDuplicates(["order_id"])

# Step 3: Convert back to DynamicFrame ONLY for output
dyf_out = DynamicFrame.fromDF(df_clean, glueContext, "output")

glueContext.write_dynamic_frame.from_options(
    frame=dyf_out,
    connection_type="s3",
    connection_options={"path": "s3://bucket/processed/"},
    format="parquet"
)
Say this in interview
"DynamicFrame is Glue's wrapper — it handles messy schemas like mixed types in the same column. DataFrame is standard PySpark — faster and has the full API including window functions. My pattern is: DynamicFrame only for reading from Glue Catalog and writing output. I convert to DataFrame immediately for all transformations. DynamicFrame for the doors, DataFrame for everything inside."
The phrase "DynamicFrame for the doors, DataFrame for everything inside" is a memorable way to explain this. Use it in interviews.
Q2 What is the Glue Catalog and how do you read from it in Spark ▼
What the Glue Catalog is
The Glue Catalog is a central metadata store — it knows the schema, location, and partitions of every table in your data lake. Think of it as a library index — it tells Spark where the books are and what is in them without Spark having to scan every shelf.
Glue CrawlerScans your S3 bucket, detects schema and partitions, and writes that metadata into the Glue Catalog automatically.
Glue CatalogStores table definitions — schema, S3 location, partition columns, file format. Used by Athena, Spark, Redshift Spectrum.
Partition pruning benefitSpark reads partition metadata from Catalog instead of listing S3 folders. Much faster — no S3 scan needed.
# Method 1: Read via Glue DynamicFrame
dyf = glueContext.create_dynamic_frame.from_catalog(
    database="analytics_db",
    table_name="orders",
    push_down_predicate="dt >= '2024-01-01'"  # filter at read time
)
df = dyf.toDF()

# Method 2: Read via Spark SQL (after configuring Glue as Hive metastore)
df = spark.read.table("analytics_db.orders") \
    .filter(col("dt") >= "2024-01-01")

# Method 3: Read only recent partitions from Glue Catalog
from pyspark.sql.functions import max as max_
df_all    = spark.read.table("analytics_db.orders")
latest_dt = df_all.agg(max_("dt")).collect()[0][0]
df_latest = df_all.filter(col("dt") == latest_dt)

# Method 4: Add new partition to Glue Catalog after writing
import boto3
glue = boto3.client("glue")
glue.create_partition(
    DatabaseName="analytics_db",
    TableName="orders",
    PartitionInput={
        "Values": ["2024-01-15"],
        "StorageDescriptor": {
            "Location": "s3://bucket/orders/dt=2024-01-15/",
            "InputFormat": "org.apache.hadoop.mapred.TextInputFormat"
        }
    }
)
Say this in interview
"The Glue Catalog is a central metadata store that holds schema, S3 location, and partition info for all tables. Glue Crawlers populate it automatically by scanning S3. When I read from Glue Catalog instead of raw S3 paths, Spark uses partition metadata directly — no S3 scan needed — which makes partition pruning much faster. I read via create_dynamic_frame.from_catalog or spark.read.table depending on whether I need DynamicFrame or not."
push_down_predicate is a powerful Glue-specific option — it filters partitions at read time before data is loaded into Spark memory. Always use it when reading large partitioned tables.
Q3 How do you handle schema evolution in Glue ▼
What schema evolution means in Glue context
Your source system adds a new column next week. Or changes a column type. Your Glue job needs to handle this without breaking — ideally without even needing a code change.
3 levels of handling — mention all three
Level 1 — CrawlerRe-run the Glue Crawler. It detects new columns and updates the Glue Catalog automatically. Set crawler to "Add new columns only" mode to avoid overwriting existing schema.
Level 2 — mergeSchemaWhen reading Parquet with mergeSchema=true, Spark unifies schemas across files. Old files return NULL for new columns.
Level 3 — Delta LakeBest for production. Delta enforces schema by default but can evolve it with mergeSchema=true. Full history of every schema change stored in transaction log.
# Level 1: Re-run crawler via boto3 (can trigger from Lambda or pipeline)
import boto3
glue = boto3.client("glue")
glue.start_crawler(Name="orders_crawler")

# Level 2: mergeSchema when reading Parquet
df = spark.read \
    .option("mergeSchema", "true") \
    .parquet("s3://bucket/orders/")
# New column in recent files → appears with NULL in older files

# Add missing expected columns defensively
expected_columns = ["order_id","user_id","amount","region","channel"]
for c in expected_columns:
    if c not in df.columns:
        df = df.withColumn(c, lit(None).cast(StringType()))

# Level 3: Delta Lake — safest evolution
df.write.format("delta") \
    .option("mergeSchema", "true") \
    .mode("append") \
    .save("s3://bucket/delta/orders/")

# Check what schema changes happened
spark.sql("""
    DESCRIBE HISTORY delta.`s3://bucket/delta/orders/`
""").select("version","timestamp","operation","operationParameters") \
    .show(truncate=False)
Say this in interview
"I handle schema evolution at three levels. First, re-run the Glue Crawler to pick up new columns in the Catalog automatically. Second, use mergeSchema when reading Parquet so Spark unifies schemas across old and new files. Third, for production pipelines I use Delta Lake — it handles all schema changes safely with a full history of what changed and when. I also add missing columns defensively in code so downstream queries never break."
The defensive column check — adding missing columns with NULL default — is a production habit. It means your pipeline never breaks even if a new column hasn't arrived yet.
Q4 How do you implement incremental processing in Glue — Job Bookmarks and partitioning ▼
Two Glue-specific tools for incremental processing
Job Bookmarks
Glue tracks which S3 files it already processed. On next run it only reads NEW files. Zero code changes — just enable in job config.
push_down_predicate
Filter partitions at read time before loading into Spark. Combines with Glue Catalog for maximum efficiency.
# ---- Job Bookmarks ----
# Enable in Glue job config:
# --job-bookmark-option = job-bookmark-enable

# In code — Glue skips already-processed files automatically
datasource = glueContext.create_dynamic_frame.from_options(
    connection_type="s3",
    connection_options={
        "paths": ["s3://bucket/raw/orders/"],
        "recurse": True
    },
    format="json",
    transformation_ctx="datasource"  # required for bookmarks
)
df = datasource.toDF()

# ---- push_down_predicate — read only recent partitions ----
from datetime import date, timedelta
cutoff = (date.today() - timedelta(7)).strftime("%Y-%m-%d")

dyf = glueContext.create_dynamic_frame.from_catalog(
    database="analytics_db",
    table_name="orders",
    push_down_predicate=f"dt >= '{cutoff}'",  # server-side filter
    transformation_ctx="orders_source"
)
df = dyf.toDF()

# ---- Combine both approaches ----
# Job Bookmarks: tracks NEW files
# push_down_predicate: reads only RECENT partitions
# Together: only new files within the last 7 days
What transformation_ctx does — important detail
The transformation_ctx is a unique string name you give each step. Glue uses it to track bookmark state per step. Without it, bookmarks do not work correctly.
Say this in interview
"Glue has two built-in tools for incremental processing. Job Bookmarks track which S3 files were already processed — enable in config, no code changes. push_down_predicate filters partitions at read time via Glue Catalog metadata — much faster than loading everything then filtering. I always add transformation_ctx to each step because Glue uses it to track bookmark state. For maximum efficiency I combine both — bookmarks for new files and push_down_predicate for recent partitions only."
transformation_ctx is a small detail that interviewers use to test whether you have actually used Glue or just read about it. Mentioning it unprompted signals real hands-on knowledge.
Q5 Reading a Glue Catalog table and converting to Spark DataFrame ▼
The full pattern — from Glue Catalog to processed output
This is a very practical question. They want to see you know the complete Glue job structure — not just one line of code.
import sys
from awsglue.transforms import *
from awsglue.utils import getResolvedOptions
from pyspark.context import SparkContext
from awsglue.context import GlueContext
from awsglue.job import Job
from awsglue.dynamicframe import DynamicFrame
from pyspark.sql.functions import col, year, avg, count

# ---- Glue job boilerplate ----
args = getResolvedOptions(sys.argv, ["JOB_NAME"])
sc   = SparkContext()
glueContext = GlueContext(sc)
spark = glueContext.spark_session
job  = Job(glueContext)
job.init(args["JOB_NAME"], args)
spark.sparkContext.setLogLevel("ERROR")

# ---- Read from Glue Catalog as DynamicFrame ----
dyf_orders = glueContext.create_dynamic_frame.from_catalog(
    database    = "analytics_db",
    table_name  = "raw_orders",
    push_down_predicate = "dt >= '2024-01-01'",
    transformation_ctx  = "raw_orders"
)

dyf_customers = glueContext.create_dynamic_frame.from_catalog(
    database   = "analytics_db",
    table_name = "customers",
    transformation_ctx = "customers"
)

# ---- Convert to DataFrame for all transforms ----
df_orders    = dyf_orders.toDF()
df_customers = dyf_customers.toDF()

# ---- Transform ----
df_joined = df_orders.join(df_customers, "customer_id", "left")

df_result = df_joined \
    .filter(col("status") == "completed") \
    .withColumn("order_year", year("order_date")) \
    .groupBy("customer_id", "order_year") \
    .agg(
        count("order_id").alias("order_count"),
        avg("amount").alias("avg_order_value")
    )

# ---- Write output ----
dyf_result = DynamicFrame.fromDF(df_result, glueContext, "result")

glueContext.write_dynamic_frame.from_options(
    frame=dyf_result,
    connection_type="s3",
    connection_options={
        "path": "s3://bucket/processed/customer_summary/",
        "partitionKeys": ["order_year"]
    },
    format="parquet",
    transformation_ctx="output"
)

# ---- Always commit the job ----
job.commit()
Say this in interview
"A Glue job always starts with the boilerplate — SparkContext, GlueContext, Job init. I read from Glue Catalog using create_dynamic_frame.from_catalog with push_down_predicate to filter partitions at read time. I convert to DataFrame immediately for all joins and transforms. Then convert back to DynamicFrame only for the output write. I always end with job.commit() — without it Glue does not update the bookmark state."
job.commit() at the end is another small detail that separates someone who has written real Glue jobs from someone who has only read documentation. Without it, Job Bookmarks never advance — every run reprocesses the same files.
Q6 Glue job writing thousands of small files — and other common Glue issues ▼
Small files — the most common Glue performance issue
Already covered in Bucket 1 Q6 but worth repeating in Glue context — Glue defaults to many small files because it uses many DPUs each writing independently.
# Fix: coalesce before writing
df.coalesce(10) \
  .write.mode("overwrite") \
  .parquet("s3://bucket/output/")

# Or use groupFiles in Glue DynamicFrame write
glueContext.write_dynamic_frame.from_options(
    frame=dyf_out,
    connection_type="s3",
    connection_options={
        "path": "s3://bucket/output/",
        "groupFiles": "inPartition",  # Glue groups small files
        "groupSize":  "134217728"     # 128MB target per file
    },
    format="parquet"
)
Other common Glue questions — quick answers
What are DPUs?Data Processing Units — Glue's unit of compute. 1 DPU = 4 vCPUs + 16GB RAM. More DPUs = more parallelism but higher cost. Standard jobs minimum 2 DPUs.
Glue vs EMRGlue is serverless — no cluster management. EMR gives full control over Spark config. Use Glue for standard ETL. Use EMR for complex tuning or ML workloads.
Glue StudioVisual ETL interface — drag and drop transforms. Generates PySpark code. Good for simple pipelines. For complex logic always write code directly.
Worker typesStandard (2 DPU), G.1X (1 DPU optimised for memory), G.2X (2 DPU for memory-intensive). Use G.1X for most jobs, G.2X for large joins.
Say this in interview
"For small files in Glue I use coalesce before writing, or the groupFiles option in Glue's write_dynamic_frame which combines small files to a target size. Glue is serverless which is great for standard ETL — no cluster management. For workloads that need fine-grained Spark tuning I'd use EMR instead. DPUs control parallelism and cost — I right-size them based on data volume."
