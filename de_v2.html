SQL Patterns
Every interview question maps to one of these 5 patterns — identify the pattern first
Pattern A Activity in a time window — "visited >3 times in 7 days", "purchased once in 30 days"
▼
The mental model
Keywords: "in last X days", "at least once in", "more than N times". The question is asking: did this user do something within a date range? Tool: WHERE date >= TODAY minus X, then GROUP BY plus HAVING.
SQL
SELECT user_id, COUNT(*) AS visits FROM visits WHERE visit_date >= CURRENT_DATE - 7 -- the window GROUP BY user_id HAVING COUNT(*) > 3; -- the threshold -- "at least once" = simpler, just use DISTINCT SELECT DISTINCT user_id FROM orders WHERE order_date >= CURRENT_DATE - 30;
PySpark
from pyspark.sql.functions import current_date, datediff, count # Visited more than 3 times in last 7 days df.filter(datediff(current_date(), col("visit_date")) <= 7) \ .groupBy("user_id") \ .agg(count("*").alias("visit_count")) \ .filter(col("visit_count") > 3) # Purchased at least once — just distinct df.filter(datediff(current_date(), col("order_date")) <= 30) \ .select("user_id").distinct()
HAVING COUNT(*) > 0 means at least once. HAVING COUNT(*) >= 3 means 3 or more. HAVING COUNT(*) > 3 means more than 3. Know the difference — interviews test this.
Pattern B Absence / inactivity — "not logged in 3 days", "not contacted in 14 days", "no order in 6 months"
▼
The mental model — two lists
List 1: ALL users. List 2: users who were ACTIVE in the window. Answer = List 1 MINUS List 2. Tool: LEFT JOIN plus IS NULL, or NOT IN subquery, or left_anti in PySpark.
SQL — LEFT JOIN + IS NULL (most common)
-- Users not logged in for last 3 days SELECT u.user_id FROM users u LEFT JOIN logins l ON u.user_id = l.user_id AND l.login_date >= CURRENT_DATE - 3 WHERE l.user_id IS NULL; -- NULL = no recent login = inactive -- Customers with no order in 6 months SELECT c.customer_id, c.name FROM customers c LEFT JOIN orders o ON c.customer_id = o.customer_id AND o.order_date >= CURRENT_DATE - 180 WHERE o.customer_id IS NULL;
PySpark — left_anti join (same logic, cleaner syntax)
from pyspark.sql.functions import datediff, current_date # Users who have NOT logged in last 3 days recent_logins = logins_df \ .filter(datediff(current_date(), col("login_date")) <= 3) \ .select("user_id").distinct() # left_anti = keep rows from LEFT with NO match on RIGHT users_df.join(recent_logins, "user_id", "left_anti")
LEFT JOIN + IS NULL equals left_anti join in PySpark. They give the exact same result. Know both — interviewers ask for both.
Pattern C Consecutive days — "logged in 3 consecutive days", "orders on consecutive days"
▼
The core trick — date minus row_number
If dates are consecutive they increase by 1 each day. Row numbers also increase by 1. So date minus row_number stays the SAME value for consecutive dates. The moment there is a gap, the value changes. Group by that value — if count is 3 or more, you have 3 consecutive days.
Example — see how the trick works
-- user | login_date | row_num | date - row_num -- U1 | 2024-01-01 | 1 | 2023-12-31 ← same group -- U1 | 2024-01-02 | 2 | 2023-12-31 ← same group -- U1 | 2024-01-03 | 3 | 2023-12-31 ← same group → 3 consecutive! -- U1 | 2024-01-07 | 4 | 2024-01-03 ← new group (gap)
Full SQL solution
WITH daily AS ( -- Step 1: dedup to one row per user per day (critical!) SELECT user_id, CAST(login_date AS DATE) AS day FROM logins GROUP BY user_id, CAST(login_date AS DATE) ), with_streak AS ( -- Step 2: calculate streak group = date minus row_number SELECT user_id, day, DATE_SUB(day, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY day) ) AS streak_group FROM daily ), streaks AS ( -- Step 3: count per streak group SELECT user_id, streak_group, COUNT(*) AS consecutive_days FROM with_streak GROUP BY user_id, streak_group ) -- Step 4: filter where streak >= N SELECT DISTINCT user_id FROM streaks WHERE consecutive_days >= 3;
PySpark
from pyspark.sql.functions import row_number, date_sub, to_date from pyspark.sql.window import Window # Step 1: dedup daily = df.withColumn("day", to_date("login_date")) \ .select("user_id","day").distinct() # Step 2+3: streak group and count w = Window.partitionBy("user_id").orderBy("day") result = daily \ .withColumn("rn", row_number().over(w)) \ .withColumn("streak_group", date_sub("day", col("rn").cast("int"))) \ .groupBy("user_id","streak_group") \ .agg(count("*").alias("consec_days")) \ .filter(col("consec_days") >= 3) \ .select("user_id").distinct()
Step 1 dedup is the most commonly forgotten step. If a user logs in twice on the same day, row numbers break and the streak calculation gives wrong results. Always dedup to date level first.

Window Functions
The most tested advanced topic — GROUP BY collapses rows, window functions keep all rows and add a column
Core formula: FUNCTION() OVER (PARTITION BY column ORDER BY column ROWS BETWEEN ...). PARTITION BY = reset per group. ORDER BY = define sequence. ROWS BETWEEN = limit the frame.
FunctionWhat it doesWhen to usePySpark
ROW_NUMBER()Unique 1,2,3 — no tiesDeduplication — keep one row per grouprow_number().over(w)
RANK()Ties same rank, gaps afterRarely used — prefer DENSE_RANKrank().over(w)
DENSE_RANK()Ties same rank, NO gapsNth highest salary, Top N per groupdense_rank().over(w)
LAG(col,n)Value from n rows beforeDay-over-day change, compare previouslag("col",1).over(w)
LEAD(col,n)Value from n rows afterGap detection, next value comparisonlead("col",1).over(w)
AVG() OVERGroup average per rowCompare each row to dept/group averageavg("col").over(w)
SUM() OVERRunning total or group sumCumulative totals, running calculationssum_("col").over(w)
NTILE(n)Divide into N equal bucketsQuartiles (4), deciles (10), percentilesntile(4).over(w)
Top Asked 2nd highest salary in each department — the most asked window function question
▼
Use DENSE_RANK with PARTITION BY dept. Wrap in CTE because you cannot filter on window function in WHERE directly.
SQL
WITH ranked AS ( SELECT name, dept, salary, DENSE_RANK() OVER ( PARTITION BY dept -- restart ranking per department ORDER BY salary DESC -- rank 1 = highest salary ) AS rnk FROM employees ) SELECT name, dept, salary FROM ranked WHERE rnk = 2; -- 2nd highest per dept
PySpark
from pyspark.sql.window import Window from pyspark.sql.functions import dense_rank w = Window.partitionBy("dept").orderBy(col("salary").desc()) df.withColumn("rnk", dense_rank().over(w)) \ .filter(col("rnk") == 2) \ .select("name","dept","salary")
Say this in interview
"I use DENSE_RANK with PARTITION BY dept and ORDER BY salary descending. DENSE_RANK handles ties correctly — if two employees share the highest salary they both get rank 1 and the next person gets rank 2. I wrap it in a CTE because you cannot filter on a window function in the same SELECT."
Running Total Running total of salaries / cumulative sum ordered by date
▼
SQL
SELECT date, sales, SUM(sales) OVER ( ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total, -- Running total that resets per month: SUM(sales) OVER ( PARTITION BY DATE_FORMAT(date,'%Y-%m') -- reset each month ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS monthly_running_total FROM daily_sales;
PySpark
from pyspark.sql.functions import sum as sum_, date_format from pyspark.sql.window import Window # Global running total w = Window.orderBy("date") \ .rowsBetween(Window.unboundedPreceding, Window.currentRow) df.withColumn("running_total", sum_("sales").over(w)) # Running total that resets per month w_month = Window.partitionBy(date_format("date","yyyy-MM")) \ .orderBy("date") \ .rowsBetween(Window.unboundedPreceding, Window.currentRow) df.withColumn("monthly_total", sum_("sales").over(w_month))
Always specify ROWS BETWEEN explicitly in PySpark. Without it, the default frame is RANGE BETWEEN which gives unexpected results with duplicate values.

PySpark Core Operations
The operations you use in every pipeline — know these cold
Core collect()[0][0] — pulling a single value out of a DataFrame
▼
PySpark never gives you a plain number directly. Everything stays as a DataFrame until you explicitly pull it out with collect(). Then index into the result like a nested list.
from pyspark.sql.functions import avg, max, min, count # Get average salary as a plain Python float avg_sal = df.agg(avg("salary")).collect()[0][0] # .agg(avg(...)) → DataFrame (a table) # .collect() → [Row(avg(salary)=66400.0)] — Python list # [0] → Row(avg(salary)=66400.0) — first row # [0] → 66400.0 — first value = your number # Get multiple aggregates — access by name (safer) result = df.agg( avg("salary").alias("avg_sal"), max("salary").alias("max_sal"), min("salary").alias("min_sal") ).collect()[0] print(result["avg_sal"]) # access by name not index position print(result["max_sal"]) # much safer if you reorder aggregations
Use collect()[0][0] for a single value. Access by name (result["col_name"]) when getting multiple values — safer than position indexing.
Core Reading Parquet, CSV, JSON from S3 — with explicit schema
▼
from pyspark.sql.types import StructType, StructField, StringType, DoubleType, TimestampType # Parquet — simplest, schema already inside the file df = spark.read.parquet("s3://bucket/data/") # CSV — ALWAYS define schema, NEVER use inferSchema in production schema = StructType([ StructField("user_id", StringType(), True), StructField("event", StringType(), True), StructField("amount", DoubleType(), True), StructField("ts", TimestampType(), True) ]) df = spark.read \ .option("header", "true") \ .option("mode", "DROPMALFORMED") \ # drop bad rows .schema(schema) \ .csv("s3://bucket/logs/") # JSON — use PERMISSIVE to capture bad rows df = spark.read \ .option("multiline", "true") \ .option("mode", "PERMISSIVE") \ .option("columnNameOfCorruptRecord", "_corrupt_record") \ .json("s3://bucket/events/")
PERMISSIVE = keeps bad rows in _corrupt_record. DROPMALFORMED = silently drops them. FAILFAST = crashes on first bad row. Production standard: PERMISSIVE then quarantine bad rows.

Bucket 1 — Reading Files
How data moves in and out of your pipeline
Top AskedHow do you handle a corrupt file in S3?
▼
from pyspark.sql.functions import col, input_file_name, current_timestamp # PERMISSIVE mode — bad rows go to _corrupt_record column df = spark.read \ .option("mode", "PERMISSIVE") \ .option("columnNameOfCorruptRecord", "_corrupt_record") \ .schema(schema) \ .json("s3://bucket/raw/events/") # Split into valid and bad valid_df = df.filter(col("_corrupt_record").isNull()) \ .drop("_corrupt_record") bad_df = df.filter(col("_corrupt_record").isNotNull()) \ .withColumn("source_file", input_file_name()) \ .withColumn("flagged_at", current_timestamp()) # Write each to separate zone valid_df.write.mode("append").parquet("s3://bucket/processed/") bad_df.write.mode("append").json("s3://bucket/quarantine/")
Say this
"PERMISSIVE mode captures bad rows in a _corrupt_record column. I split into valid and quarantine DataFrames — valid goes to processed, bad goes to quarantine with source filename and timestamp so the team can investigate. I alert if bad rate crosses 5%."
Top AskedSmall files problem — Glue job writing thousands of small files
▼
# Problem: Spark writes one file per partition — can be hundreds of tiny files df.write.parquet("s3://bucket/output/") # 500 small files # Fix 1: coalesce — reduces partitions WITHOUT full shuffle (cheaper) df.coalesce(8).write.parquet("s3://bucket/output/") # Fix 2: repartition — full shuffle, guaranteed even distribution df.repartition(8).write.parquet("s3://bucket/output/") # Fix 3: Delta Lake OPTIMIZE — compacts files automatically spark.sql("OPTIMIZE delta.`s3://bucket/delta/orders/`")
Say this
"Small files happen because Spark keeps too many partitions after filtering. I use coalesce before writing — no full shuffle so it is cheaper. Target is 128 to 256MB per file. For Delta Lake I run OPTIMIZE periodically to compact small files automatically."
2️⃣
Bucket 2 — Performance
Skew, joins, cache, partitioning — the most asked advanced PySpark topic
Top AskedSkewed data — how to detect and fix it
▼
Skew means one partition has far more data than others — like 80% of rows having country=US. That one task runs for hours while others finish in minutes.
# Step 1: detect skew df.groupBy(spark_partition_id().alias("partition")) \ .count().orderBy("count", ascending=False).show() # Fix 1: broadcast join (when one table is small — under 50MB) from pyspark.sql.functions import broadcast result = df_large.join(broadcast(df_small), "country") # Fix 2: salting (both tables large) from pyspark.sql.functions import rand, concat, lit, array, explode SALT = 10 df_large = df_large \ .withColumn("salt", (rand()*SALT).cast("int").cast("string")) \ .withColumn("salted_key", concat(col("country"),lit("_"),col("salt"))) df_small = df_small \ .withColumn("salts", array([lit(str(i)) for i in range(SALT)])) \ .withColumn("salt", explode("salts")) \ .withColumn("salted_key", concat(col("country"),lit("_"),col("salt"))) \ .drop("salts","salt") result = df_large.join(df_small, "salted_key").drop("salted_key","salt")
Say this
"Skew means one partition has far more data — like 80% of rows being country equals US. I detect it by checking partition sizes. Fix: broadcast join if one table is small, simplest option. If both tables are large I use salting — add a random number to the join key so the hot key spreads across multiple partitions."
ImportantCache vs Persist — the difference and when to use
▼
from pyspark import StorageLevel # cache() = shorthand for persist(MEMORY_AND_DISK) df.cache() # persist() = control the storage level df.persist(StorageLevel.MEMORY_ONLY) # fastest, OOM risk df.persist(StorageLevel.MEMORY_AND_DISK) # spills to disk if needed df.persist(StorageLevel.DISK_ONLY) # slow, for huge DataFrames # Real usage pattern df_active = df.filter(col("status") == "active").cache() count = df_active.count() # TRIGGERS caching (cache is lazy!) agg1 = df_active.groupBy("dept").count() # reuses cache df_active.unpersist() # FREE MEMORY when done
Say this
"cache() is shorthand for persist with MEMORY_AND_DISK level. persist() gives control to choose the storage level. I use cache when a DataFrame is used multiple times in the same job to avoid recomputing from scratch. Important: cache is lazy so I always call .count() right after to trigger it. And I always unpersist when done to free memory."

Bucket 3 — Incremental Processing
Process only NEW data — not everything every time
Top AskedHow do you implement incremental processing — 3 approaches
▼
# Approach 1: partition filter — simplest from datetime import date, timedelta today = date.today().strftime("%Y-%m-%d") df = spark.read.parquet("s3://bucket/events/") \ .filter(col("dt") == today) # Approach 2: watermark in DynamoDB — for tables with updated_at import boto3 dynamodb = boto3.resource("dynamodb") table = dynamodb.Table("pipeline_watermarks") def get_watermark(job): return table.get_item( Key={"job_name":job}).get("Item",{}).get("last_ts","1970-01-01") def save_watermark(job, ts): table.put_item( Item={"job_name":job,"last_ts":str(ts)}) last_ts = get_watermark("orders_pipeline") df = spark.read.parquet("s3://bucket/orders/") \ .filter(col("updated_at") > last_ts) # Process, write... new_ts = df.agg(max_("updated_at")).collect()[0][0] save_watermark("orders_pipeline", new_ts) # save AFTER successful write # Approach 3: Glue Job Bookmarks # Enable in Glue job config — Glue tracks processed files automatically # Add transformation_ctx to every step — required for bookmarks to work
Say this
"Three approaches: partition filter for date-partitioned S3 data, watermark stored in DynamoDB for tables with an updated_at column, or Glue Job Bookmarks where AWS manages state automatically. I always save the watermark AFTER a successful write — not before. If I save before and the write fails, those records are skipped forever."
ImportantHow do you handle late arriving data?
▼
# Streaming — watermarking df_stream.withWatermark("event_time", "30 minutes") \ .groupBy(window("event_time","1 hour"), "user_id") \ .count() # Spark keeps window open 30 minutes past the window end # Late events arriving within 30 min are included # Batch — reprocess last N days with partition overwrite from datetime import date, timedelta reprocess_from = (date.today() - timedelta(3)).strftime("%Y-%m-%d") df = spark.read.parquet("s3://bucket/events/") \ .filter(col("event_date") >= reprocess_from) # OVERWRITE mode — safe to reprocess, idempotent df.write.mode("overwrite") \ .partitionBy("event_date") \ .parquet("s3://bucket/processed/")
Say this
"In streaming I use watermarking — tell Spark to wait 30 minutes for late events before closing a window. In batch I reprocess the last 3 days using an updated_at column and write with partition overwrite. The overwrite pattern makes the job idempotent — running it twice gives the same result."

Bucket 4 — Data Quality
Detect, tag, preserve, alert — never silently drop bad data
Top AskedThe 4 quality layers — what every pipeline needs
▼
Layer 1 SchemaDoes the data have the right columns and types? Validate at read time with explicit schema definition.
Layer 2 CompletenessAre required fields NULL? A record with no customer_id cannot be joined to anything — flag it.
Layer 3 Business rulesDoes the data make sense? Salary cannot be negative. Age must be 0 to 120. Order date cannot be in the future.
Layer 4 Volume/statsDid the expected amount of data arrive? If yesterday had 100k rows and today has 500 — something broke upstream.
def run_quality_checks(df): total = df.count() checks = { "null_user_ids": df.filter(col("user_id").isNull()).count(), "null_amounts": df.filter(col("amount").isNull()).count(), "negative_amount": df.filter(col("amount") < 0).count(), "future_dates": df.filter(col("order_date") > current_date()).count(), } for check, cnt in checks.items(): status = "PASS" if cnt == 0 else "FAIL" print(f"[{status}] {check}: {cnt}/{total}") return checks
Say this
"I think about data quality in four layers: schema validation at read time, completeness checks for required fields, business rule checks like salary cannot be negative, and volume checks to catch upstream failures. Each layer catches different problems and together they give me confidence the data is trustworthy."

Bucket 5 — AWS Glue
DynamicFrame for the doors, DataFrame for everything inside
Top AskedDynamicFrame vs DataFrame — and the full Glue job pattern
▼
import sys from awsglue.utils import getResolvedOptions from pyspark.context import SparkContext from awsglue.context import GlueContext from awsglue.job import Job from awsglue.dynamicframe import DynamicFrame args = getResolvedOptions(sys.argv, ["JOB_NAME"]) sc = SparkContext() glueContext = GlueContext(sc) spark = glueContext.spark_session job = Job(glueContext) job.init(args["JOB_NAME"], args) # Step 1: Read as DynamicFrame (handles messy schemas) dyf = glueContext.create_dynamic_frame.from_catalog( database="my_db", table_name="raw_orders", push_down_predicate="dt >= '2024-01-01'", # filter at read time transformation_ctx="raw_orders" # REQUIRED for bookmarks ) # Step 2: Convert to DataFrame IMMEDIATELY for all transforms df = dyf.toDF() df = df.filter(col("status") == "active") \ .withColumn("order_year", year("order_date")) # Step 3: Convert back to DynamicFrame ONLY for output dyf_out = DynamicFrame.fromDF(df, glueContext, "output") glueContext.write_dynamic_frame.from_options( frame=dyf_out, connection_type="s3", connection_options={"path":"s3://bucket/processed/"}, format="parquet", transformation_ctx="output" ) job.commit() # CRITICAL — without this, bookmarks never advance
Say this
"DynamicFrame for the doors, DataFrame for everything inside. I read from Glue Catalog as DynamicFrame, convert to DataFrame immediately for all transforms, then convert back only for the output write. Two things I always do: add transformation_ctx to every step so bookmarks track state correctly, and always end with job.commit() — without it bookmarks never advance and every run reprocesses all files."

Bucket 6 — Lambda
Lambda detects and delegates — never the heavy processing engine
Core PatternLambda cold start, timeout, and the optimised production pattern
▼
import boto3, json, os # OUTSIDE handler — initialised ONCE on cold start, reused on warm calls s3_client = boto3.client("s3") glue_client = boto3.client("glue") sns_client = boto3.client("sns") JOB_NAME = os.environ["GLUE_JOB_NAME"] # from env variables SNS_TOPIC = os.environ["SNS_TOPIC_ARN"] def lambda_handler(event, context): # Check remaining time (15 min hard limit) remaining = context.get_remaining_time_in_millis() try: bucket = event["Records"][0]["s3"]["bucket"]["name"] key = event["Records"][0]["s3"]["object"]["key"] if not key.startswith("raw/orders/"): # validate input return {"status": "skipped"} # Lambda's role: detect trigger, delegate to Glue response = glue_client.start_job_run( JobName=JOB_NAME, Arguments={"--input_path": f"s3://{bucket}/{key}"} ) return {"status":"triggered","job_run_id":response["JobRunId"]} except Exception as e: sns_client.publish(TopicArn=SNS_TOPIC, Subject="Lambda failure", Message=str(e)) raise # re-raise so CloudWatch sees failure AND alarm fires
Say this
"Lambda has a 15-minute hard limit so I never do heavy data processing in Lambda. Lambda's role is to detect events and delegate — it spots a new S3 file and immediately triggers a Glue job, then returns. I always move client initialisations outside the handler so they run once and reuse on warm calls. And I re-raise after sending the SNS alert so CloudWatch also sees the failure — otherwise CloudWatch thinks the job succeeded."

Bucket 7 — Delta Lake
Parquet files plus _delta_log folder — that log is everything
Top AskedDelta Lake vs Parquet — and the MERGE upsert pattern
▼
# Delta Lake = Parquet files + _delta_log/ folder # That transaction log gives: ACID, MERGE, time travel, schema evolution # Write Delta df.write.format("delta").mode("overwrite") \ .partitionBy("year","month") \ .save("s3://bucket/delta/orders/") # MERGE — upsert: update if exists, insert if new (atomic) from delta.tables import DeltaTable target = DeltaTable.forPath(spark, "s3://bucket/delta/orders/") target.alias("t").merge( updates_df.alias("s"), "t.order_id = s.order_id" # match condition ).whenMatchedUpdateAll() \ # update all columns if match .whenNotMatchedInsertAll() \ # insert all if new .execute() # Time travel — query data as it was in the past df_old = spark.read.format("delta") \ .option("timestampAsOf","2024-01-14") \ .load("s3://bucket/delta/orders/") # Restore after bad job run dt = DeltaTable.forPath(spark, "s3://bucket/delta/orders/") dt.restoreToVersion(12) # one line rollback
Say this
"Delta Lake is not a new file format — it is Parquet files plus a _delta_log folder. That log records every change and gives ACID transactions, MERGE for upserts, and time travel to query past versions. Plain Parquet for simple append-only pipelines. Delta when I need to update records, handle late corrections, or implement SCD Type 2."
8️⃣
Bucket 8 — Pipeline Reliability
Idempotent, quarantine, restore, prevent
Top AskedHow do you handle a job that fails halfway through?
▼
# Key word: idempotent — running twice gives same result as once # Technique 1: partition overwrite — always safe to retry df.write.mode("overwrite") \ .partitionBy("event_date") \ .parquet("s3://bucket/processed/") # Rerunning overwrites same partition = no duplicates # Technique 2: Delta MERGE — handles re-runs naturally target.alias("t").merge(source.alias("s"), "t.id = s.id") \ .whenMatchedUpdateAll().whenNotMatchedInsertAll().execute() # Running MERGE twice with same data = same result # Technique 3: quarantine bad records, continue pipeline valid_df = df.filter(col("_corrupt").isNull()) bad_df = df.filter(col("_corrupt").isNotNull()) valid_df.write.mode("append").parquet("s3://processed/") bad_df.write.mode("append").json("s3://quarantine/")
Say this
"The key is idempotency — designing the pipeline so re-running it gives the same result as running it once. I achieve this with partition overwrite mode so writing the same partition twice does not create duplicates, or with Delta Lake MERGE which naturally handles re-runs. For corrupted input I use PERMISSIVE mode — bad records go to quarantine, valid records continue, the job does not fail because of a few bad rows."

System Design Foundations
10 core terms and the 5-layer framework — every system design question uses these
Must KnowThe 5-step framework — say this structure every single time
▼
Step 1 ClarifyAsk questions BEFORE drawing anything. Is this batch or real-time? What is the data volume? Who are the consumers? What is the SLA for freshness?
Step 2 EstimateDo the math out loud. 500k vehicles times 1 event per 30 seconds equals 17,000 events per second. This determines tools.
Step 3 DesignWalk through 5 layers: Ingestion → Storage → Processing → Serving → Monitoring.
Step 4 Trade-offsWhy this tool not that one? Batch vs streaming? Cost vs latency? What changes at 10x scale?
Step 5 FailuresWhat if job fails halfway? Corrupt file? Late data? Always address these — it shows production experience.
Open every system design answer with this
"Before I design anything, let me clarify a few things — is this batch or real-time? What is the data volume per day? Who are the downstream consumers — analysts, dashboards, or ML models? And what is the acceptable latency for data freshness?"
Never start drawing architecture without asking at least 2 clarifying questions. Interviewers deliberately leave requirements vague to see if you ask. Jumping straight to an answer is a red flag.
10 Key TermsData lake, warehouse, lakehouse, ETL vs ELT, SLA, idempotency, lineage and more
▼
Data LakeCentral S3 repository storing raw data in original format. Never delete raw data. Source of truth you can always reprocess from.
Data WarehouseStructured, query-optimised store — Redshift, Snowflake. Pre-aggregated, schema-enforced. Fast for dashboards.
Data LakehouseCombination — S3 storage like a lake, but with warehouse features via Delta Lake or Apache Iceberg. Best of both worlds.
ETL vs ELTETL: Transform before loading (old way). ELT: Load raw then transform in warehouse (modern — S3 → Redshift → dbt). ELT is more flexible.
SLAService Level Agreement — the deadline for data freshness. "Dashboard must show data within 1 hour of event" is an SLA. Drives batch vs streaming decision.
IdempotencyRunning the pipeline twice gives the same result. Critical for reliability. Achieved with overwrite mode or Delta MERGE.
LineageTracking where data came from and what transformed it. "This dashboard number came from this raw file." Used for debugging and compliance.
Schema-on-read vs writeSchema-on-write: enforce when writing (Delta, Redshift). Schema-on-read: apply when reading (raw S3 CSV). Lake uses read, warehouse uses write.
OLAP vs OLTPOLTP: operational database, many small transactions (MySQL). OLAP: analytical queries, large scans (Redshift, Athena). DE works with OLAP.
Lambda ArchitectureBatch layer for historical accuracy + streaming layer for real-time + merged at serving. Most production systems use both.

from pyspark.sql.window import Window w_7d = Window.partitionBy("vehicle_id") \ .orderBy("timestamp") \ .rowsBetween(-7*24*120, 0) # 7 days of 30-sec readings df = df \ .withColumn("avg_temp_7d", avg("engine_temp").over(w_7d)) \ .withColumn("max_temp_7d", max("engine_temp").over(w_7d)) \ .withColumn("risk_score", when((col("avg_temp_7d") > 95) & (col("max_temp_7d") > 100), "HIGH") .when(col("avg_temp_7d") > 90, "MEDIUM") .otherwise("LOW"))

from pyspark.sql.window import Window from pyspark.sql.functions import lag, sum as sum_, when # New session if gap > 30 minutes w = Window.partitionBy("user_id").orderBy("event_time") df = df \ .withColumn("gap_mins", (col("event_time").cast("long") - lag("event_time").over(w).cast("long"))/60) \ .withColumn("new_session", when(col("gap_mins") > 30, 1).otherwise(0)) \ .withColumn("session_id", sum_("new_session").over( w.rowsBetween(Window.unboundedPreceding, 0))) # Funnel analysis funnel = df.groupBy("dt").agg( countDistinct(when(col("event")=="page_view","user_id")).alias("visitors"), countDistinct(when(col("event")=="add_to_cart","user_id")).alias("add_to_cart"), countDistinct(when(col("event")=="purchase","user_id")).alias("purchased") )

Most Asked System Design Questions
These come up in almost every DE system design interview
Very CommonHow would you migrate from on-premise to AWS?
▼
Phase 1 ReplicateAWS DMS or DataSync replicates on-premise data to S3 without changing anything. Both systems run in parallel.
Phase 2 Process on AWSMove ETL jobs to Glue while still reading from replicated S3 data. Compare outputs to on-premise results exactly.
Phase 3 Serve on AWSMove dashboards and queries to Athena or Redshift. Validate results match on-premise reports.
Phase 4 Cut overStop on-premise jobs. AWS becomes primary. Keep on-premise available for 30 days as rollback option.
Say this
"Phased migration — never big bang. First replicate data to S3, then move processing to Glue while running both in parallel, then move serving to Athena. Only cut over when outputs match exactly. Keep on-premise available as rollback for 30 days. The key is never having a day where both systems are down simultaneously."
