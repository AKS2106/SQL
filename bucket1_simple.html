Q1 Parquet vs CSV vs JSON — differences and when to use each ▼
Understand this first — before the code
Parquet
Columnar binary format. Schema stored inside. Spark reads only columns it needs. 3–5x smaller than CSV. Default choice.
CSV
Plain text rows. No schema stored — Spark guesses it. Reads all columns every time. Largest size. Use for external system exports.
JSON
Text key-value pairs. Handles nested and flexible schemas. Slower due to parsing. Use for APIs and varied schemas.
The key insight to mention — columnar reading
If your query needs only 2 columns from a 50-column table — Parquet reads only those 2 off disk. CSV reads all 50 and throws 48 away. That is why Parquet is 10–20x faster for analytics.
Code
from pyspark.sql.types import *

# Parquet — simplest, schema already inside the file
df = spark.read.parquet("s3://bucket/data/")

# CSV — always define schema yourself, never let Spark guess
schema = StructType([
    StructField("user_id", StringType(),    True),
    StructField("event",   StringType(),    True),
    StructField("amount",  DoubleType(),    True),
    StructField("ts",      TimestampType(), True)
])
df = spark.read \
    .option("header", "true") \
    .option("mode", "DROPMALFORMED") \
    .schema(schema) \
    .csv("s3://bucket/logs/")

# JSON — multiline if one record spans multiple lines
df = spark.read \
    .option("multiline", "true") \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .json("s3://bucket/events/")
Say this in interview
"Parquet is my default for any analytical workload — it's columnar so Spark only reads the columns it needs, and schema is embedded so no inference needed. CSV when receiving data from external systems. JSON when schema is nested or varies per record."
Never use inferSchema=True on CSV in production. Spark reads the entire file twice and can guess wrong — a column with mostly numbers but one "N/A" row becomes StringType.
Q2 Read modes — PERMISSIVE, DROPMALFORMED, FAILFAST ▼
What are "bad rows"
A bad row is one Spark cannot parse — wrong type, missing columns, or broken JSON. These three modes tell Spark what to do when it hits one.
PERMISSIVE (default)Keeps bad rows. Puts the raw broken text into a column called _corrupt_record. Good fields are parsed, bad ones become NULL. Use this when you want to inspect what went wrong.
DROPMALFORMEDSilently drops bad rows. No error, no record. Use when bad rows are noise — but risky because you don't know how many were dropped.
FAILFASTThrows an exception on the very first bad row and stops the job. Use for strict pipelines where any bad row means something is seriously wrong upstream.
# PERMISSIVE — capture bad rows in _corrupt_record column
df = spark.read \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .schema(schema) \
    .csv("s3://bucket/logs/")

df.filter(col("_corrupt_record").isNotNull()).show()

# DROPMALFORMED — silently ignore bad rows
df = spark.read.option("mode","DROPMALFORMED").schema(schema) \
    .csv("s3://bucket/logs/")

# FAILFAST — crash immediately on first bad row
df = spark.read.option("mode","FAILFAST").schema(schema) \
    .csv("s3://bucket/logs/")
Say this in interview
"I always use PERMISSIVE in development — it shows me what bad data looks like. In production I use PERMISSIVE with a quarantine write so bad rows are preserved for investigation. FAILFAST for strict financial pipelines where one bad row means something is seriously wrong upstream."
DROPMALFORMED is tempting but dangerous — you have no idea how many rows were silently lost. Always at minimum count and log the bad rows.
Q3 Corrupt file in S3 — separate valid and bad records (quarantine pattern) ▼
The quarantine pattern — production standard
Never silently drop bad records. Write them to a quarantine folder with metadata so someone can investigate and reprocess later.
from pyspark.sql.functions import col, input_file_name, current_timestamp

# Step 1: Read with PERMISSIVE
df = spark.read \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .schema(schema) \
    .json("s3://bucket/raw/events/")

# Step 2: Split into valid and bad
valid_df = df.filter(col("_corrupt_record").isNull()) \
             .drop("_corrupt_record")

bad_df = df.filter(col("_corrupt_record").isNotNull()) \
           .withColumn("source_file", input_file_name()) \
           .withColumn("ingested_at", current_timestamp())

# Step 3: Log counts — always know your bad rate
total   = df.count()
bad_cnt = bad_df.count()
print(f"Bad rows: {bad_cnt}/{total} = {100*bad_cnt/total:.1f}%")

# Step 4: Write valid to processed zone
valid_df.write.mode("append") \
    .partitionBy("event_date") \
    .parquet("s3://bucket/processed/")

# Step 5: Write bad to quarantine zone
bad_df.write.mode("append").json("s3://bucket/quarantine/")

# Step 6: Alert if bad rate is too high
if total > 0 and bad_cnt / total > 0.05:
    import boto3
    boto3.client("sns").publish(
        TopicArn="arn:aws:sns:us-east-1:123:alerts",
        Message=f"Bad record rate: {bad_cnt}/{total}",
        Subject="Data quality alert"
    )
Say this in interview
"I use PERMISSIVE mode to capture bad rows, split the DataFrame into valid and bad, write valid to processed and bad to quarantine with source filename and timestamp. I also alert if bad rate crosses 5%."
The source_file and ingested_at columns on the bad DataFrame are what makes this production-grade. Without them you can't trace where the bad data came from.
Q4 Large CSV in Python without loading all data into memory ▼
Why it is a problem
pd.read_csv("10gb_file.csv") loads all 10GB into RAM. If your machine has 8GB — it crashes. Solution: read and process one chunk at a time.
import pandas as pd

chunk_size = 100_000   # 100k rows in memory at a time
results = []

for chunk in pd.read_csv("large_file.csv", chunksize=chunk_size):
    # Only this chunk is in memory right now
    filtered = chunk[chunk["status"] == "active"]
    agg = filtered.groupby("dept")["salary"].sum()
    results.append(agg)

# Combine all chunk results at the end
final = pd.concat(results).groupby(level=0).sum()
print(final)
When to use PySpark instead
# PySpark — no chunking needed, Spark handles distribution
df = spark.read.option("header","true").schema(schema) \
    .csv("s3://bucket/large_file.csv")
df.filter(col("status")=="active") \
  .groupBy("dept").agg(sum("salary")).show()
Say this in interview
"In Python I use pandas chunksize — process 100k rows at a time so only that chunk is in memory. For files over 5GB I move to PySpark — no chunking needed since Spark distributes automatically across executors."
They will ask: "When pandas vs PySpark?" — pandas for files under 5GB on one machine. PySpark for larger files or when you need distributed processing.
Q5 Schema changes in Parquet — add column, type change, removed column ▼
3 types of changes — each handled differently
New column addedSafe. Old files return NULL for the new column. Use mergeSchema=true when reading so Spark unifies both schemas.
Column type changedRisky. Cast explicitly. Or use Delta Lake which handles type evolution safely.
Column removedOld files still have it. New files don't. Usually safe — monitor downstream queries that may depend on it.
# New column — mergeSchema unifies old and new file schemas
df = spark.read \
    .option("mergeSchema", "true") \
    .parquet("s3://bucket/data/")
# Old files: new_col = NULL  |  New files: new_col = real value

# Type changed — cast it explicitly after reading
df = df.withColumn("price", col("price").cast(DoubleType()))

# Safest for ongoing schema changes — use Delta Lake
df.write.format("delta") \
    .option("mergeSchema", "true") \
    .mode("append") \
    .save("s3://bucket/delta/orders/")
Say this in interview
"Adding a column is safe — old files return NULL with mergeSchema. Type changes I handle with explicit casting. For production pipelines with frequent schema changes I use Delta Lake — it handles all three cases safely with a full history of what changed."
Q6 Glue job writing thousands of small files — how to fix it ▼
Why it happens
Spark writes one file per partition. After a filter or join Spark keeps many partitions even though much less data remains. 500 partitions × 1MB each = 500 small files. Reading them is slow — every file needs a separate S3 request.
The fix — reduce partitions before writing
# Problem: too many tiny files
df.write.parquet("s3://bucket/output/")

# Fix 1: coalesce — reduces partitions, NO full shuffle (cheaper)
df.coalesce(8).write.parquet("s3://bucket/output/")

# Fix 2: repartition — full shuffle, perfectly even distribution
# Use when coalesce gives uneven file sizes
df.repartition(8).write.parquet("s3://bucket/output/")

# Fix 3: repartition by column — best for partitioned output
df.repartition(8, "dept") \
  .write.partitionBy("dept") \
  .parquet("s3://bucket/output/")

# Fix 4: Delta Lake OPTIMIZE — compacts files automatically
spark.sql("OPTIMIZE delta.`s3://bucket/delta/orders/`")
coalesce vs repartition — know both
coalesce(N)No full shuffle. Moves data locally. Faster and cheaper. May give slightly uneven files. Use this first.
repartition(N)Full shuffle across cluster. Perfectly even. More expensive. Use when data is skewed or coalesce gives uneven results.
Say this in interview
"Small files happen because Spark keeps too many partitions after filtering. I use coalesce before writing — no full shuffle so it's cheaper. Target is 128 to 256MB per file. For Delta Lake I run OPTIMIZE periodically to compact files automatically."
Rule of thumb: 128–256MB per output file. For 2GB of data coalesce(10) gives ~200MB files. Always check actual file sizes in S3 after writing.
