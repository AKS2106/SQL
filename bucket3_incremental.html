Bucket 3 — Incremental Processing

5 questions. The core idea of this entire bucket is one thing: how do you process only NEW data instead of reprocessing everything every time. Once you understand that, every question here becomes easy.
Q1 What is incremental processing — and why does it matter ▼
The core idea — full load vs incremental
Imagine you have 1 year of order data — 100 million rows. Every day 50,000 new orders arrive. You have two choices:
Full load — bad
Every day read all 100M rows, reprocess everything, write everything again.

Cost: high. Time: hours. Wasteful — 99.95% of data didn't change.
Incremental — good
Every day read only the 50,000 NEW rows. Process and append them.

Cost: tiny. Time: minutes. Processes only what changed.
The 3 approaches — know all three
1. Partition filterData is partitioned by date in S3. Read only today's folder. Simplest approach — no state needed.
2. Watermark / timestampStore the last processed timestamp. Next run reads only rows where updated_at > last timestamp. Works for any table with an updated_at column.
3. Glue Job BookmarksAWS manages the state for you. Glue tracks which S3 files were already processed. Enable in Glue config — no code changes needed.
Say this in interview
"Incremental processing means reading only new or changed data instead of reprocessing everything. There are three approaches: partition filter for date-partitioned S3 data, watermark tracking using an updated_at column, and Glue Job Bookmarks where AWS manages the state automatically. I choose based on how the data is structured."
Q2 How do you implement incremental processing — with code ▼
Approach 1 — partition filter (simplest)
If your S3 data is partitioned by date, just read today's partition. No state to track. No watermark. Just filter by today's date.
from datetime import date, timedelta
from pyspark.sql.functions import col

# Read only today's partition
today = date.today().strftime("%Y-%m-%d")
df = spark.read.parquet("s3://bucket/orders/") \
    .filter(col("dt") == today)

# Or read last N days (useful for catching late data)
n_days_ago = (date.today() - timedelta(3)).strftime("%Y-%m-%d")
df = spark.read.parquet("s3://bucket/orders/") \
    .filter(col("dt") >= n_days_ago)
Approach 2 — watermark with DynamoDB (production pattern)
Store the last processed timestamp somewhere persistent (DynamoDB, S3, or a parameter store). On each run, read only rows after that timestamp. After a successful run, save the new timestamp.
import boto3
from pyspark.sql.functions import col, max as max_

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("pipeline_watermarks")

# --- Read last watermark ---
def get_watermark(job_name):
    resp = table.get_item(Key={"job_name": job_name})
    return resp.get("Item", {}).get("last_ts", "1970-01-01")

# --- Save new watermark ---
def save_watermark(job_name, last_ts):
    table.put_item(Item={
        "job_name": job_name,
        "last_ts":  str(last_ts)
    })

# --- Use in pipeline ---
last_ts = get_watermark("orders_pipeline")
print(f"Processing records after: {last_ts}")

# Read only new/updated rows
df = spark.read.parquet("s3://bucket/orders/") \
    .filter(col("updated_at") > last_ts)

# Process...
df_processed = df.withColumn(...)

# Write output
df_processed.write.mode("append").parquet("s3://bucket/processed/")

# Save new watermark — only after successful write
new_ts = df.agg(max_("updated_at")).collect()[0][0]
save_watermark("orders_pipeline", new_ts)
Approach 3 — Glue Job Bookmarks
Glue keeps track of which S3 files it has already processed. Enable it in the Glue job config. No code changes needed at all.
# Enable in Glue job config (AWS Console or Terraform):
# Job Bookmarks = ENABLED

# Or pass as job argument:
# --job-bookmark-option = job-bookmark-enable

# In your Glue script — no changes needed
# Glue automatically skips already-processed files
datasource = glueContext.create_dynamic_frame.from_options(
    connection_type="s3",
    connection_options={"paths": ["s3://bucket/raw/orders/"]},
    format="json"
)
# Glue only reads NEW files since last successful run
Say this in interview
"For date-partitioned S3 data I use a simple partition filter — read only today's folder. For tables with an updated_at column I use a watermark stored in DynamoDB — read rows after last timestamp, process, then save the new timestamp only after a successful write. For Glue batch jobs the simplest option is Job Bookmarks — Glue tracks which files were processed with no code changes needed."
Always save the watermark AFTER a successful write — not before. If you save it before and the write fails, those records are skipped forever.
Q3 What happens if Glue Job Bookmarks are NOT enabled ▼
The simple answer
Without Job Bookmarks, Glue has no memory of what it processed before. Every time the job runs it reads ALL files in the S3 path from the very beginning — not just the new ones.
Day 1 run
→
Reads 10 files
→
Writes 10k rows
Day 2 run
→
Reads ALL 11 files again
→
Writes 11k rows (duplicates!)
Day 3 run
→
Reads ALL 12 files again
→
More duplicates
The result
Your output table gets more and more duplicate data every day. The job also gets slower every run because it's reading more and more files. Eventually what was a 5-minute job becomes a 2-hour job.
How to fix if you already have duplicates
# Option 1: deduplicate output after enabling bookmarks
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number

w = Window.partitionBy("order_id").orderBy(col("updated_at").desc())
df_deduped = df.withColumn("rn", row_number().over(w)) \
               .filter(col("rn") == 1) \
               .drop("rn")

# Option 2: Delta Lake MERGE — upsert handles duplicates cleanly
from delta.tables import DeltaTable
target = DeltaTable.forPath(spark, "s3://bucket/delta/orders/")
target.alias("t").merge(
    df_deduped.alias("s"),
    "t.order_id = s.order_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()
Say this in interview
"Without Job Bookmarks Glue has no memory — every run reads all files from the beginning, creating duplicates in the output and getting slower every day. The fix is to enable bookmarks going forward, and deduplicate existing data using ROW_NUMBER or Delta Lake MERGE."
This is the answer to a very specific question from your list. It's a short answer — problem is duplicates and growing cost, fix is enable bookmarks plus deduplicate.
Q4 How do you handle late arriving data ▼
What late arriving data means
An event happens at 9pm but due to network delays, app crashes, or slow APIs — it arrives in your pipeline at 11pm or even the next day. Your pipeline already processed the 9pm window and moved on. What do you do with this late record?
In streaming pipelines
Use watermarking. Tell Spark to keep a window open for X minutes/hours waiting for late data before closing it.
In batch pipelines
Reprocess the last N days every run using updated_at column. Overwrite those partitions so late records get included.
Streaming — watermarking in Spark Structured Streaming
# Watermark tells Spark: "wait up to 30 minutes for late events"
# After 30 min past the window end — close it, ignore later arrivals

df_stream = spark.readStream.format("kinesis")...

windowed = df_stream \
    .withWatermark("event_time", "30 minutes") \
    .groupBy(
        window("event_time", "1 hour"),  # 1-hour window
        "user_id"
    ).count()

# Spark keeps each window open for 1hr + 30min before closing
# Events arriving up to 30 min late are included correctly
Batch — reprocess last N days (most practical)
from datetime import date, timedelta
from pyspark.sql.functions import col

# Reprocess last 3 days to catch any late arrivals
reprocess_from = (date.today() - timedelta(3)).strftime("%Y-%m-%d")

df = spark.read.parquet("s3://bucket/events/") \
    .filter(col("event_date") >= reprocess_from)

# Process the data
df_processed = df.withColumn(...)

# Write with OVERWRITE on partition — this is idempotent
# Running it twice gives the same result
df_processed.write \
    .mode("overwrite") \
    .partitionBy("event_date") \
    .parquet("s3://bucket/processed/")
SQL pipeline — using updated_at column
-- Instead of filtering by event_date, filter by when record was updated
-- This catches records that arrived late and were inserted late

SELECT *
FROM orders
WHERE updated_at >= CURRENT_DATE - 3   -- last 3 days of changes
ORDER BY updated_at DESC;
Say this in interview
"In streaming I use watermarking — tell Spark to wait 30 minutes for late events before closing a window. In batch I reprocess the last 3 days using an updated_at column and write with partition overwrite — so late records get included and the result is the same whether I run it once or ten times. That overwrite pattern also makes the job idempotent."
The word idempotent is important to use here. It means running the job twice gives the same result. Overwrite mode on partitions achieves this — interviewers look for this word specifically.
Q5 Reading only the latest partition from S3 or Glue Catalog ▼
Why this matters
If your S3 bucket has 2 years of daily partitions — 730 folders — and you only need today's data, you don't want Spark scanning all 730 folders. You want it to read only the latest one.
Method 1 — dynamically find the latest partition
from pyspark.sql.functions import col, max as max_

# Read all data, find latest partition value
all_df  = spark.read.parquet("s3://bucket/events/")
latest  = all_df.agg(max_("dt")).collect()[0][0]
print(f"Latest partition: {latest}")

# Now read only that partition — Spark skips all other folders
df = all_df.filter(col("dt") == latest)
Method 2 — use Glue Catalog (most efficient)
Glue Catalog stores partition metadata — it knows which partitions exist without scanning S3. This is the fastest approach.
# Glue Catalog knows partitions without scanning S3
df = spark.read.table("glue_db.events")
latest = df.agg(max_("dt")).collect()[0][0]
df_latest = df.filter(col("dt") == latest)
Method 3 — boto3 list S3 folders directly
import boto3

s3 = boto3.client("s3")
response = s3.list_objects_v2(
    Bucket="my-bucket",
    Prefix="events/",
    Delimiter="/"   # list top-level folders only
)

# Get all partition folders, sort, take last one
partitions = [p["Prefix"] for p in response["CommonPrefixes"]]
latest_prefix = sorted(partitions)[-1]
print(f"Latest: {latest_prefix}")  # events/dt=2024-01-15/

df = spark.read.parquet(f"s3://my-bucket/{latest_prefix}")
Regex — extract date from file path (from your question list)
from pyspark.sql.functions import input_file_name, regexp_extract

df = spark.read.parquet("s3://bucket/events/")

# Add the file path as a column
df = df.withColumn("file_path", input_file_name())

# Extract date parts using regex
# Path: s3://bucket/events/year=2024/month=01/day=15/file.parquet
df = df \
    .withColumn("year",
        regexp_extract("file_path", r"year=(\d{4})", 1)) \
    .withColumn("month",
        regexp_extract("file_path", r"month=(\d{2})", 1)) \
    .withColumn("day",
        regexp_extract("file_path", r"day=(\d{2})", 1))

# Path: s3://bucket/logs/2024-01-15_events.json
df = df.withColumn("date_str",
    regexp_extract("file_path", r"(\d{4}-\d{2}-\d{2})", 1))
Say this in interview
"I find the latest partition value first then filter on it — Spark skips all other folders, which is partition pruning. Using Glue Catalog is the most efficient because it has partition metadata indexed already so it doesn't need to scan S3 at all. For extracting dates from file paths I use input_file_name() with regexp_extract."
Partition pruning = Spark reads only matching folders and skips everything else. This is the term interviewers want to hear when you talk about partition-based filtering.
