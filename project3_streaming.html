Concept 1 What is streaming — batch vs streaming explained simply ▼
The simplest way to understand the difference
Batch (Projects 1 and 2)
Data arrives, you wait until a lot has collected, then process it all at once. Like waiting to do laundry until the basket is full.

Delay: minutes to hours.
Use: daily reports, ETL jobs.
Streaming (Project 3)
Data arrives continuously, you process each event as it comes in. Like washing one item of clothing the moment it gets dirty.

Delay: seconds.
Use: fraud detection, IoT alerts, dashboards.
Real example from your CX background
In CX you might get a daily report showing customer complaints from yesterday — that is batch. Streaming is getting an alert the moment a customer complaint comes in so you can act on it within seconds. Same data, completely different timing.
The three components of any streaming pipeline
ProducerGenerates events and sends them to the stream. In this project: a Python script simulating vehicle sensors sending readings every second.
Message brokerHolds the events in order and lets consumers read them. In this project: Kafka running in Docker. On AWS this would be Kinesis.
ConsumerReads events from the broker and processes them. In this project: Spark Structured Streaming reading from Kafka, detecting anomalies.

Concept 2 What is Kafka — and what is Docker ▼
Kafka in plain English
Kafka is a message queue. Producers write messages to a topic (like a named channel). Consumers read from that topic. Kafka holds messages in order and multiple consumers can read the same messages independently.
Key Kafka terms you need to know
TopicA named channel. Like a WhatsApp group — producers send to it, consumers read from it. In this project: "vehicle-sensors" topic.
PartitionTopics are split into partitions for parallelism. More partitions = more throughput. We use 3 partitions.
OffsetPosition of a message in a partition. Like a page number. Consumers track their offset so they know what they have already read.
Consumer groupMultiple consumers sharing the work. Each partition is read by one consumer in the group.
Docker in plain English
Docker runs applications in isolated containers — like a lightweight virtual machine. Instead of installing Kafka and its dependencies on your laptop (complicated), you run a Docker container that already has everything set up. One command and Kafka is running.
Without Docker
Install Java, install Zookeeper, configure Zookeeper, install Kafka, configure Kafka, set environment variables... 2 hours of setup, often breaks.
With Docker
docker-compose up

One command. Kafka running in 30 seconds. Stop it when done — nothing installed on your machine permanently.
Concept 3 What is windowing and watermarking — the two key streaming concepts ▼
Windowing — group events by time
In batch you group by a column value. In streaming you group by time windows — collect all events in the last 2 minutes and aggregate them. This is a window.
Tumbling windowFixed non-overlapping windows. 9:00-9:02, 9:02-9:04, 9:04-9:06. Each event belongs to exactly one window. Used in this project.
Sliding windowOverlapping windows. Every 30 seconds, look at last 2 minutes. More granular but more compute.
Watermarking — handle late arriving data
A sensor reading from 9:01 might arrive at 9:05 due to network delay. Without watermarking Spark has already closed the 9:00-9:02 window and the late event is lost. Watermarking tells Spark: keep the window open for X minutes to wait for late events.
# Without watermark — late events lost
df.groupBy(window("event_time", "2 minutes"))

# With watermark — wait up to 1 minute for late events
df.withWatermark("event_time", "1 minute") \
  .groupBy(window("event_time", "2 minutes"))
In interviews: "I use withWatermark to handle late-arriving sensor data — tells Spark to hold each window open for an extra minute before closing it. Without it, late events from slow sensors would be silently dropped."

Step 3 Write the producer — simulate vehicle sensor data ▼
What the producer does
Simulates 100 vehicles each sending sensor readings every second. 5% of readings are intentionally anomalous — high engine temperature or high RPM. The consumer will detect these.
Create src\producer.py
import json
import time
import random
from datetime import datetime
from kafka import KafkaProducer

# ── Connect to Kafka ─────────────────────────────────────────────
producer = KafkaProducer(
    bootstrap_servers="localhost:9092",
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    key_serializer=lambda k: k.encode("utf-8")
)

# ── 100 simulated vehicles ───────────────────────────────────────
VEHICLES = [f"VH{str(i).zfill(4)}" for i in range(1, 101)]

def generate_reading(vehicle_id):
    """Generate one sensor reading for a vehicle.
    5% chance of anomaly — high temp or high RPM."""

    is_anomaly = random.random() < 0.05   # 5% anomaly rate

    if is_anomaly:
        anomaly_type = random.choice(["high_temp", "high_rpm", "low_battery"])
        if anomaly_type == "high_temp":
            engine_temp = round(random.uniform(102, 115), 1)
            rpm         = round(random.gauss(2800, 300))
            battery_v   = round(random.uniform(12.5, 14.0), 2)
        elif anomaly_type == "high_rpm":
            engine_temp = round(random.gauss(90, 5), 1)
            rpm         = round(random.uniform(5200, 6500))
            battery_v   = round(random.uniform(12.5, 14.0), 2)
        else:  # low_battery
            engine_temp = round(random.gauss(88, 5), 1)
            rpm         = round(random.gauss(2800, 300))
            battery_v   = round(random.uniform(10.5, 11.8), 2)
    else:
        # Normal reading
        engine_temp = round(random.gauss(88, 4), 1)
        rpm         = round(random.gauss(2800, 400))
        battery_v   = round(random.uniform(13.2, 14.4), 2)

    return {
        "vehicle_id":   vehicle_id,
        "event_time":   datetime.utcnow().isoformat(),
        "engine_temp":  engine_temp,
        "rpm":          max(500, rpm),     # min 500 RPM
        "battery_v":    battery_v,
        "speed_kmh":    round(random.uniform(0, 120)),
        "fuel_pct":     round(random.uniform(5, 100)),
        "is_anomaly":   is_anomaly         # ground truth for testing
    }

# ── Send events continuously ─────────────────────────────────────
print("=" * 50)
print("  VEHICLE SENSOR PRODUCER STARTED")
print(f"  Vehicles: {len(VEHICLES)}")
print(f"  Anomaly rate: 5%")
print(f"  Topic: vehicle-sensors")
print("=" * 50)
print("Sending... (Ctrl+C to stop)\n")

event_count = 0

try:
    while True:
        # Send readings for 10 random vehicles each second
        batch = random.sample(VEHICLES, 10)
        for vehicle_id in batch:
            reading = generate_reading(vehicle_id)
            producer.send(
                topic="vehicle-sensors",
                key=vehicle_id,       # partition key — same vehicle → same partition
                value=reading
            )
            event_count += 1

        producer.flush()

        # Print summary every 10 seconds
        if event_count % 100 == 0:
            print(f"  Sent {event_count:,} events so far...")

        time.sleep(1)   # 10 events per second

except KeyboardInterrupt:
    print(f"\n✓ Producer stopped. Total events sent: {event_count:,}")
    producer.close()

Step 4 Write the consumer — Spark Structured Streaming reads from Kafka ▼
Install the Kafka-Spark connector

Create src\consumer.py
from pyspark.sql import SparkSession
from pyspark.sql.functions import (
    from_json, col, avg, max as max_,
    count, window, current_timestamp,
    when, round as round_
)
from pyspark.sql.types import (
    StructType, StructField,
    StringType, DoubleType,
    IntegerType, BooleanType
)

# ── Create Spark session with Kafka connector ────────────────────
spark = SparkSession.builder \
    .appName("VehicleAnomalyDetector") \
    .master("local[*]") \
    .config("spark.sql.shuffle.partitions", "4") \
    .config("spark.jars.packages",
            "org.apache.spark:spark-sql-kafka-0-10_2.12:3.4.1") \
    .getOrCreate()

spark.sparkContext.setLogLevel("ERROR")

# ── Schema for the JSON payload ──────────────────────────────────
schema = StructType([
    StructField("vehicle_id",  StringType(),  True),
    StructField("event_time",  StringType(),  True),
    StructField("engine_temp", DoubleType(),  True),
    StructField("rpm",         IntegerType(), True),
    StructField("battery_v",   DoubleType(),  True),
    StructField("speed_kmh",   DoubleType(),  True),
    StructField("fuel_pct",    DoubleType(),  True),
    StructField("is_anomaly",  BooleanType(), True)
])

print("=" * 55)
print("  VEHICLE ANOMALY DETECTOR — SPARK STREAMING")
print("=" * 55)
print("Connecting to Kafka topic: vehicle-sensors...")
print("Waiting for events... (Ctrl+C to stop)\n")

# ── Read stream from Kafka ───────────────────────────────────────
raw_stream = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "vehicle-sensors") \
    .option("startingOffsets", "latest") \
    .load()

# ── Parse JSON payload ───────────────────────────────────────────
# Kafka delivers raw bytes — cast to string, parse JSON
df_parsed = raw_stream \
    .withColumn("data",
        from_json(col("value").cast("string"), schema)) \
    .select("data.*") \
    .withColumn("event_ts",
        col("event_time").cast("timestamp"))

# ── Anomaly detection — 2 minute tumbling window ─────────────────
# For each vehicle in each 2-minute window:
# Calculate avg temp, max RPM, min battery
# Flag if any reading exceeds safe thresholds

windowed = df_parsed \
    .withWatermark("event_ts", "1 minute") \
    .groupBy(
        window(col("event_ts"), "2 minutes"),   # 2-min tumbling window
        col("vehicle_id")
    ).agg(
        round_(avg("engine_temp"), 1).alias("avg_temp"),
        max_("engine_temp").alias("max_temp"),
        max_("rpm").alias("max_rpm"),
        round_(avg("battery_v"), 2).alias("avg_battery"),
        count("*").alias("readings")
    )

# ── Add alert reason ─────────────────────────────────────────────
alerts = windowed.withColumn("alert",
    when(col("max_temp") > 100,    "HIGH_TEMP")
    .when(col("max_rpm")  > 5000,  "HIGH_RPM")
    .when(col("avg_battery") < 12, "LOW_BATTERY")
    .otherwise(None)
).filter(col("alert").isNotNull())   # only show anomalies

# ── Stream 1: Print ALL windowed results to console ──────────────
query_all = windowed.writeStream \
    .outputMode("update") \
    .format("console") \
    .option("truncate", False) \
    .option("numRows", 5) \
    .trigger(processingTime="10 seconds") \
    .start()

# ── Stream 2: Print ONLY alerts to console ───────────────────────
query_alerts = alerts \
    .select(
        col("window.start").alias("window_start"),
        col("vehicle_id"),
        col("alert"),
        col("avg_temp"),
        col("max_rpm"),
        col("avg_battery"),
        col("readings")
    ) \
    .writeStream \
    .outputMode("update") \
    .format("console") \
    .option("truncate", False) \
    .queryName("ANOMALY_ALERTS") \
    .trigger(processingTime="10 seconds") \
    .start()

# ── Wait for streams ─────────────────────────────────────────────
print("Streams started. Results appear every 10 seconds.")
print("You need the producer running in a separate terminal.")
print("Ctrl+C to stop.\n")

spark.streams.awaitAnyTermination()
The consumer has two output streams. Stream 1 prints all windowed aggregations so you can see data flowing. Stream 2 prints only vehicles with anomalies — these are your alerts. In production Stream 2 would write to DynamoDB or trigger an SNS notification.

# Vehicle Anomaly Detection — Real-Time Streaming Pipeline

Real-time IoT streaming pipeline that simulates sensor data from
100 vehicles, detects engine anomalies using Spark Structured
Streaming, and surfaces alerts within seconds of occurrence.

## Business Problem
A fleet company monitors 100 vehicles in real-time. Engine failures
cost £5,000+ per breakdown. Detecting anomalies 10 minutes early
allows proactive maintenance and prevents roadside incidents.
This pipeline processes 10 sensor events/second and surfaces
alerts within a 2-minute window.

## Architecture
Vehicle sensors → Python producer → Kafka (Docker)
→ Spark Structured Streaming consumer
→ 2-minute tumbling windows
→ Anomaly alerts (HIGH_TEMP / HIGH_RPM / LOW_BATTERY)

In production: Kafka → AWS Kinesis, Local → AWS EMR

## What This Pipeline Detects
| Alert | Condition | Meaning |
|---|---|---|
| HIGH_TEMP | Engine temp > 100°C in window | Overheating risk |
| HIGH_RPM | RPM > 5000 in window | Engine strain |
| LOW_BATTERY | Avg battery < 12V | Charging failure |

## Key Technical Decisions

**Kafka as message broker**
Kafka decouples producers from consumers — vehicles keep sending
data even if the consumer crashes and resumes. On AWS this would
be replaced by Kinesis Data Streams.

**2-minute tumbling window with 1-minute watermark**
Single readings can spike briefly. Aggregating over 2 minutes
reduces false positives. Watermark handles sensor data that
arrives up to 1 minute late due to network delays.

**vehicle_id as partition key**
Routing all readings from the same vehicle to the same partition
ensures readings arrive in order per vehicle — important for
sequential anomaly detection.

**Checkpoint location (production note)**
In production I would add:
.option("checkpointLocation", "s3://bucket/checkpoints/vehicles/")
This lets the streaming job resume from exactly where it left off
if it crashes — no data loss, no duplicate processing.
